## Block 1 — 3 응용 강의 (40분)

### 1. 자율주행 — Latency 우선

- 워크로드: Perception (CNN) + Planning (작은 GEMM) + Control
- 우선: **Latency p99** (frame 33ms 이내) — 안전 제약. p99 critical (worst case = 사고)
- batch=1 (한 frame 씩)
- 단순화: ResNet-style GEMM 시퀀스 [(1, 2048, 2048), (1, 2048, 1000), ...]

### 2. LLM 챗봇 — Throughput per Cost

- 워크로드: Prefill (큰 GEMM) + Decode (batch=1 GEMV × 100 steps)
- 우선: **Tokens/sec/$**
- Prefill 은 큰 GEMM → util 좋음. Decode 는 batch=1 → util 나쁨
- 단순화: prefill (256, 4096, 4096) + decode (256, 4096, 1) × 100 steps

### 3. 영상처리 (엣지) — Throughput per Watt

- 워크로드: 카메라 → MobileNet-class CNN
- 우선: **TOPS/W** (전력 예산 1-10W)
- 단순화: 작은 GEMM 시퀀스 [(1, 256, 256)] × 20 + [(1, 256, 1000)]

### 응용별 우선 metric 표

| 응용 | 우선 metric | 부속 제약 |
| --- | --- | --- |
| 자율주행 | latency p99 | 33ms 이내 |
| LLM 챗봇 | tokens/sec/$ | latency p50 < 100ms/token |
| 영상처리 | TOPS/W | <5W |

## Block 2 — 응용별 워크로드 모델링 (40분)

### 학생 과제 (바이브 코딩)

> ```python
> def workload_autonomous() -> list[tuple[int, int, int]]:
>     return [(1, 2048, 2048), (1, 2048, 2048), (1, 2048, 1024), (1, 1024, 512), (1, 512, 1000)]
>
> def workload_llm_chat(batch=256) -> dict:
>     return {'prefill': (batch, 4096, 4096), 'decode': (batch, 4096, 1), 'decode_steps': 100}
>
> def workload_video_edge() -> list[tuple[int, int, int]]:
>     return [(1, 256, 256)] * 20 + [(1, 256, 1000)]
> ```

### 본인 NPU 측정

각 워크로드를 본인 `NPU.run` 으로 실행. 우선 metric 계산:
- 자율주행: 시퀀스 전체 latency_ms (frame time)
- 챗봇: tokens/sec = batch × decode_steps / total_time. $/token = cost / tokens (cost = 4-2 면적 × 단가)
- 영상: TOPS/W = peak util 시점의 활성 throughput / 평균 전력

### 검증 기준

- 자율주행 frame time 1ms ~ 100ms 범위
- 챗봇 prefill 의 PE util 50%+ (PE 크기에 따라)
- 영상 GEMM 이 작아서 underutilized (PE 차원이 256 보다 크면)
- 적어도 1 응용에서 본인 *defining feature* 가 우위로 나타남

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 챗봇 decode 너무 느림 | batch=1 GEMV systolic underutilized | 정상. continuous batching 을 4-4 에서 도입 |
| 영상 util 이 0 에 가까움 | 작은 GEMM 이 큰 PE 못 채움 | 정상. PE 크기 줄이는 것이 4-4 옵션 |
| 자율주행 frame time 너무 짧음 | flush 단계 카운트 누락 | 3-3 simulator flush 점검 |
| TOPS/W 가 무한대 | static power 누락 | 4-2 정적 전력 |
| 모든 응용에서 본인이 win | 비교 칩 reuse 낮음 또는 cost 무시 | Week 2-4 기준 사용 |
