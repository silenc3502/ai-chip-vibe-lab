## Part A. 3 응용 강의 (60분)

### 1. 자율주행 — 안전과 latency

#### 워크로드 패턴
- **Perception**: 카메라/LiDAR → CNN (ResNet-class) → 객체 검출
- **Planning**: 검출 결과 → 의사결정 (작은 GEMM + control logic)
- **Control**: planning → actuator (실시간성 critical)

#### 우선 metric
- **Latency** (frame 단위, 33ms 또는 16ms 이내) — 안전 제약
- p99 latency가 더 중요 (worst case가 사고를 일으킴)
- batch=1 (한 frame씩, 누적 안 됨)

#### 단순화 워크로드
- "ResNet-50의 일부": Conv가 많은 layer 시퀀스
- 본 회차에서는 GEMM 시퀀스로 환원: [(1, 2048, 2048), (1, 2048, 1000), ...]

### 2. LLM 챗봇 — Throughput per Cost

#### 워크로드 패턴
- **Prefill**: 입력 프롬프트의 모든 토큰을 한번에 처리 (큰 GEMM)
- **Decode**: 토큰 1개씩 생성 (작은 GEMM, batch=1, KV cache lookup)
- 서비스 레벨에서는 *수많은 사용자*의 동시 요청을 batch로 묶음

#### 우선 metric
- **Throughput per $**: 데이터센터 단위 비용당 토큰/초
- Prefill은 큰 GEMM이라 PE utilization 좋음
- Decode는 batch=1이라 utilization 나쁨 (큰 batch로 묶기 위해 *continuous batching* 등 기법)

#### 단순화 워크로드
- Prefill: (256, 4096, 4096) — batch=256, hidden=4096
- Decode: (256, 1, 4096) → (256, 4096, 1) GEMV
- 합쳐서 시간 합산

### 3. 영상처리 (엣지) — Throughput per Watt

#### 워크로드 패턴
- 카메라 → DNN (object detection / 분류) → action
- *온디바이스 (mobile, IoT)* → 전력 매우 critical
- 스트리밍 (frame이 계속 들어옴)

#### 우선 metric
- **Throughput per Watt** (TOPS/W)
- *전력 budget* 보통 1~10W (노트북 NPU = ~5W, 폰 = ~1W)

#### 단순화 워크로드
- MobileNet-class: 작은 conv 시퀀스 (depthwise + pointwise 교대)
- 본 회차에서는 GEMM 시퀀스로 환원: [(1, 256, 256), (1, 256, 256), ...] (작은 GEMM 다수)

### 응용별 우선 metric 표

| 응용 | 우선 metric | 부속 제약 | 본 회차 워크로드 |
| --- | --- | --- | --- |
| 자율주행 | latency p99 | 33ms 이내 | GEMM 시퀀스 (큰 layer, batch=1) |
| LLM 챗봇 | tokens/sec/$ | latency p50 < 100ms/token | Prefill (큰 batch) + Decode (batch=many) |
| 영상처리 | TOPS/W | <5W | GEMM 시퀀스 (작은 layer, batch=1) |

## Part B. 응용별 워크로드 모델링 + 본인 NPU 측정 (90분)

### 학생 과제 (바이브 코딩으로)

> 다음 함수 3개를 작성:
>
> ```python
> def workload_autonomous() -> list[tuple[int, int, int]]:
>     """ResNet-style GEMM sequence, batch=1"""
>     return [(1, 2048, 2048), (1, 2048, 2048), (1, 2048, 1024), (1, 1024, 512), (1, 512, 1000)]
>
> def workload_llm_chat(batch: int = 256) -> dict:
>     """returns prefill + decode workload"""
>     return {
>         'prefill': (batch, 4096, 4096),
>         'decode': (batch, 4096, 1),  # GEMV per batch
>         'decode_steps': 100,
>     }
>
> def workload_video_edge() -> list[tuple[int, int, int]]:
>     """MobileNet-class small GEMM sequence, batch=1"""
>     return [(1, 256, 256)] * 20 + [(1, 256, 1000)]
> ```
>
> 각 워크로드를 본인 NPU.run으로 실행. 응용별 우선 metric 계산:
> - 자율주행: 시퀀스 전체의 latency_ms (frame time)
> - 챗봇: tokens/sec (=batch * decode_steps / total_time), 그리고 $/token (cost는 4-2 면적 모델로)
> - 영상: TOPS/W (peak utilization 시점의)

### 검증 기준

- 자율주행 frame time이 합리적 범위 (1ms ~ 100ms)
- 챗봇 prefill GEMM이 큰 utilization (PE array 크기에 따라 50%+)
- 영상 GEMM이 작아서 underutilized (PE 차원이 256보다 크면)
- 적어도 1개 응용에서 본인 NPU의 *defining feature*가 우위로 나타남

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 챗봇 decode가 너무 느림 | batch=1 GEMV가 systolic underutilized | 정상 — 그게 챗봇의 진짜 어려움. *continuous batching*을 4-4에서 도입 가능 |
| 영상 워크로드의 utilization이 0에 가까움 | 작은 GEMM이 큰 PE array를 못 채움 | 정상. *PE array 크기 줄이는 것*이 4-4 옵션 |
| 자율주행 frame time이 너무 짧음 | latency 모델이 partial sum 수집 끝나는 시간을 카운트 안 함 | 3-3 simulator의 flush 단계 점검 |
| TOPS/W 계산이 무한대 | static power 누락 | 4-2의 정적 전력 더하기 |
| 모든 응용에서 본인이 win | 비교 칩 reuse_factor가 낮음 또는 cost 무시 | Week 2-4 기준 사용 |
