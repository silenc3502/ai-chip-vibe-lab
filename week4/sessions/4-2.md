## Block 1 — Latency vs Throughput + Cost 정의 (40분)

### Latency vs Throughput

| metric | 정의 | 단위 |
| --- | --- | --- |
| Latency | 한 query 시작→완료 | ms |
| Throughput | 단위 시간 처리 query 수 | QPS, tokens/sec |
| Tail (p99) | 99th percentile | ms |

**핵심 통찰**: *대립하는 metric*. 응용에 따라 *어느 끝* 선택:
- Batch=1: 짧은 latency, 낮은 throughput
- Batch=N: 긴 latency, 높은 throughput

### Cost 요소

| 요소 | 추정 | 비고 |
| --- | --- | --- |
| 다이 면적 | Σ 블록별 면적 | mm². 7nm ~$0.2/mm² |
| 전력 | Σ 동적 + 정적 | W |

> 본 회차는 *면적 + 전력* 만. 패키징/HBM 단순화.

### Performance per Cost

- TOPS/$ — 가격당 throughput
- TOPS/W — 전력당 throughput (효율)
- Tokens/sec/$ — LLM 응용
- Latency × $ — 자율주행 (둘 다 작아야)

## Block 2 — Batching Curve 측정 (40분)

### 학생 과제 (바이브 코딩)

> 본인 NPU 시뮬로 sweep:
> - Batch: [1, 2, 4, 8, 16, 32, 64, 128, 256, 512]
> - 워크로드: GEMM (batch, 4096, 4096) — LLM 류
>
> 각 batch 에 latency_ms, throughput = batch / latency. plot: latency vs batch, throughput vs batch.

### 검증 기준

- batch=1: latency 작음, throughput 낮음
- batch < pe_rows: latency *거의 평탄*, throughput 정비례 증가
- knee point (batch ≈ pe_rows): util 1.0 도달
- knee 이후: latency 정비례, throughput saturate
- 그래프 패턴: ↑ → ↗

### Knee Point 분석

1. throughput 이 saturate 시작하는 batch?
2. 그 batch 의 latency? (응용 latency 제약과 비교)
3. *latency-친화* batch 와 *throughput-친화* batch 각 1개

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| Throughput 이 batch=1 부터 saturate | utilization 이 batch 무관 | input batch 가 PE 의 한 차원을 채우게 |
| Latency 가 batch 와 정비례 | 동시 처리 모델링 누락 | 한 batch *전체*가 한 systolic 사이클 그룹 |
| 다이 면적 1 mm² 같은 비현실 | 단위 혼동 | TPU v4 ~600 mm² sanity check |
| Static power 누락 | util=0 에 전력 0 | static = peak × 0.1 |
