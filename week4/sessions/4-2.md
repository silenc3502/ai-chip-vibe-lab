## Part A. 성능 축 정의 (45분)

### Latency vs Throughput

| metric | 정의 | 단위 |
| --- | --- | --- |
| Latency | 한 query의 *시작 → 응답 완료* 시간 | ms |
| Throughput | 단위 시간당 처리 가능한 query 수 | QPS (queries/sec) 또는 tokens/sec |
| Tail latency (p99) | 99th percentile latency | ms |

**핵심 통찰**: Latency와 Throughput은 *같은 metric의 다른 면*이 아니라 *대립하는 metric*.

- Batch=1: 짧은 latency, 낮은 throughput (PE underutilized)
- Batch=N: 긴 latency, 높은 throughput (PE well utilized)

→ 응용에 따라 *어느 끝*을 선택할지 다름.

### Cost — 칩의 가격을 결정하는 것들

| 요소 | 추정 모델 | 비고 |
| --- | --- | --- |
| 다이 면적 | Σ (블록별 면적) | mm². 7nm에서 ~$0.2/mm² 단가 |
| 전력 (TDP) | Σ (블록별 동적 + 정적) | W |
| 패키징 | 면적과 핀 수 | $1~50 |
| HBM | 용량 × $/GB | $5~20/GB |

> 본 회차에서는 *다이 면적*과 *전력*만 다룬다. 패키징/메모리는 단순화.

### Performance per Cost — 1D 메트릭으로 환원

- **TOPS/$**: 칩 단위 가격당 throughput
- **TOPS/W**: 전력당 throughput (효율)
- **Tokens/sec/$ for LLM**: LLM 추론 응용
- **Latency × $**: 자율주행 응용 (latency × 비용 모두 작아야)

## Part B. Batching Curve 측정 (75분)

### 학생 과제 (바이브 코딩으로)

> 본인 NPU 시뮬레이터로 다음 sweep:
>
> - 배치 크기: [1, 2, 4, 8, 16, 32, 64, 128, 256, 512]
> - 워크로드: GEMM (batch_size, 4096, 4096) — LLM-like
>
> 각 batch에 대해 다음 측정:
> - latency_ms (한 batch 처리 시간)
> - throughput_qps = batch / latency
>
> Plot: x축 batch (log), y축 latency, 그 다음 throughput.

### 검증 기준

- batch=1: latency 작음 (예: 1-10ms), throughput 낮음
- batch가 PE array 차원 미만일 때: latency *거의 평탄* (PE는 underutilized이지만 streaming 시간은 동일), throughput은 batch에 정비례 증가
- knee point (보통 batch ≈ pe_r): PE array 첫 fill, util 1.0 도달
- knee 이후: latency가 batch에 *정비례* 증가, throughput은 saturate
- 즉 그래프는 "↑ → ↗" (knee 전 flat, 후 선형)

### Knee Point 분석

학생은 다음 답해야:

1. throughput이 saturate 시작하는 batch는?
2. 그 batch에서 latency는 얼마? (응용 latency 제약과 비교에 쓸 수치)
3. 본인 NPU의 *latency-friendly batch*는 무엇이고, *throughput-friendly batch*는?

학생 과제:

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| Throughput이 batch=1부터 saturate | utilization 모델이 batch와 무관 | batch가 PE rows에 매핑되도록 — input batch가 PE의 한 차원을 채우게 |
| Latency가 batch와 정비례 | 동시 처리 모델링 누락 | 한 batch *전체*가 한 systolic 사이클 그룹 안에 들어가게 |
| 다이 면적이 1 mm² 같은 비현실적 작은 값 | area_per_PE 또는 SRAM 단가 단위 혼동 | 표 단위 다시 확인, TPU v4 환산이 ~600 mm² 나오는지 sanity check |
| Static power 누락 | utilization=0이어도 전력 0 | static_power = peak × 0.1 더하기 |
