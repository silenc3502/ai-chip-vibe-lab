## Block 1 — 발표 준비 + 슬라이드 점검 (40분)

### 발표 템플릿 (8-10분)

| 시간 | 슬라이드 | 내용 |
| --- | --- | --- |
| 0:00-1:00 | 1. 문제 | 어느 응용? 우선 metric? |
| 1:00-2:30 | 2. 출발점 | Defining feature, base 핵심 결정 3개 |
| 2:30-4:00 | 3. Week 1-2 도구 | Roofline, dataflow — 본인이 어떻게 썼나 |
| 4:00-5:30 | 4. Week 3 NPU | 시뮬 결과 + **RTL/cocotb 검증** |
| 5:30-7:30 | 5. Week 4 튜닝 | 변경 + 결과 + specialization 비용 |
| 7:30-9:00 | 6. 한 줄 평 | *"___ 에 ___ 만큼 좋고, 대가는 ___"* |
| 9:00-10:00 | 7. 다음 단계 | 더 해 본다면? |

### 슬라이드 점검 체크리스트

- [ ] 각 슬라이드 *수치* 1개 이상
- [ ] *Defining feature* 가 슬라이드 1, 5 에 일관되게
- [ ] *Trade-off* 명시 (모든 게 좋아짐 X)
- [ ] 그래프/표 1개 이상
- [ ] **RTL ↔ Python 검증 결과 한 줄** (Week 3-2 또는 3-4)

### 발표 팁

- 시간 *준수*. 8분 안에 끝낼 수 있게 연습
- 첫 30초 = *문제* 부터 (구조 X)
- 마지막 30초 = *한 줄 평* 천천히

### 발표 (이어서)

남은 학생 발표.

### 피어 리뷰 (15분)

- 본인이 못 들은 학생 *1명 이상*
- 이상적: *다른 응용* 선택한 학생

### 리뷰 양식 (`peer_review_<발표자>.md`)

```markdown
# 피어 리뷰: <발표자>의 <응용/NPU 이름>

## 1. 가장 인상적인 발견 (2-3 문장)
## 2. *Defining feature → 수치 우위* 번역 설득력
## 3. Trade-off 인정의 정직성 (회귀 정직 보고?)
## 4. 더 알고 싶은 것 1개 (질문)
## 5. 한 줄 평 — "이 발표가 인상적인 이유는 ___, 더 강해질 부분은 ___"
```

> 200~400자. *비판이 아니라 발전점* 위주.

## Block 4 — 코스 회고 + 강사 마무리 (40분)

### `course_retrospective.md` 작성 (25분)

```markdown
# 80시간 코스 회고

## 1. 가장 큰 변화
- 코스 시작 vs 끝 — *AI 칩을 보는 방식* 이 어떻게 달라졌나?

## 2. 가장 어려웠던 회차
- 어느 회차? 무엇이 어려웠나? 어떻게 극복?

## 3. 가장 surprising 한 발견
- 측정 / 시뮬 / RTL ↔ Python 비교 / 토의 중 1개

## 4. 바이브 코딩 + RTL 도구 회고
- 가장 잘 작동한 프롬프트 패턴
- 가장 빗나간 프롬프트 패턴
- Python 과 Verilog 작성 시 프롬프트 패턴이 어떻게 달랐나?
- 80시간 후 본인의 프롬프트 작성 방식 변화

## 5. 본인 NPU 설계의 정직 평가
- 4주차 끝 NPU 가 *얼마나 좋은가*?
- 가장 약점은? (시뮬 정확도, 설계 결정, 측정 신뢰도, RTL 한계)

## 6. 다음 단계
- 더 시간이 있다면? (전체 systolic RTL? FPGA 포팅? 실제 칩 평가?)
- 본인 분야에 어떻게 적용?

## 7. 코스 피드백
- 가장 가치 있었던 부분
- 개선 제안
```

### 그룹 토의 (10분)

3-4명 그룹: Q1 (가장 큰 변화) 와 Q3 (가장 surprising) 공유. *공통 surprise* 발견.

### 강사 마무리 멘트 (5분, 제안)

> *"우리는 4주간 진짜 칩을 fab 에 보낸 적은 없습니다. 그러나 RTL 을 작성하고, cocotb 로 검증하고, Python 모델과 결과 일치를 확인했습니다. 이게 진짜 칩 설계자가 거치는 흐름의 *첫 두 단계* 입니다 — architectural model + RTL 시뮬. 다음 단계는 합성 / 배치 / fab 인데, 그건 도구만 다르고 *사고 방식은 같습니다*."*
>
> *"바이브 코딩 도구는 4주간 코드 작성 속도를 N배 빠르게 했습니다. 도구가 자동화될 때 사람의 일은 *무엇을 만들지* 와 *왜* 를 명확히 하는 것 — 프롬프트 로그가 그 증거입니다. 같은 도구로 Python 도 Verilog 도 cocotb 도 — 도구가 변해도 *왜* 는 사람이 정해야 합니다."*

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 발표 10분 넘김 | 슬라이드 3-4 너무 김 | 1줄 압축, 자세한 건 보고서 |
| 발표가 *결과* 에서 막힘 | 측정 신뢰도 낮음 | 정직하게 *"이 부분은 시뮬/RTL 한계"* — 그게 좋은 발표 |
| 피어 리뷰 모호 (*"좋았어요"*) | 측정/trade-off 관점 부족 | *수치* 코멘트 강제 |
| 회고 *"재미있었어요"* 수준 | 7 섹션 안 채움 | 최소 Q1, Q5, Q6 강제 |
| 자기 NPU 자신감 잃음 | 다른 학생이 인상적 | 정상. *비교가 아니라 본인이 무엇을 배웠는가* |
