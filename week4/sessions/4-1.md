## Block 1 — Multi-Objective + Pareto 이론 (40분)

### Single vs Multi-objective

- Single: *"가장 빠른 NPU"* — 답은 한 점
- Multi: *"빠르면서 작고 전력도 적은"* — 답은 *집합* (Pareto front)

### Pareto-dominated 정의

A 가 B 를 *dominate* ↔ 모든 metric 에서 A ≥ B 이고 하나 이상에서 A > B.
**Pareto front** = 누구에게도 dominated 되지 않는 config 집합.

### 흔한 metric

| metric | 방향 | 단위 |
| --- | --- | --- |
| latency | 작을수록 | ms |
| energy | 작을수록 | mJ |
| 다이 면적 | 작을수록 | mm² |
| TOPS | 클수록 | TOPS |
| TOPS/W | 클수록 | TOPS/W |

> 본 회차는 (latency, energy) 2D. 4-2 에서 cost 추가.

## Block 2 — DSE Sweep 실험 (40분)

### 학생 과제 (바이브 코딩)

> 본인 NPUConfig 베이스로 3축 sweep:
>
> - SRAM: [1, 4, 16, 64] MB (4단계)
> - PE array: [(8,8), (16,16), (32,32), (64,64)] (4단계)
> - dtype: [int8, fp16, fp32] (3단계)
>
> 총 **4×4×3 = 48 config**. 각 config 로 워크로드 (W4 = 1024³ 등 작게) 시뮬, (latency_ms, energy_mJ) 수집. matplotlib scatter, base config 다른 색.

### 검증 기준

- 48 점이 plot 에 모두
- 평면을 *덮음* (한 클러스터에 모이면 sweep range 좁음)
- INT8 config 가 일반적으로 좌하단 (빠르고 에너지 적음)
- FP32 가 우상단 또는 비효율
- 본인 base config 위치 명시

### 함정

- 시뮬 너무 느림 → 워크로드 작게 (1024³ 또는 512³)
- 모든 점이 한 점에 모임 → sweep 효과가 시뮬에 반영 안 됨 (3-5 회귀)
