# 4-4. 응용 맞춤 튜닝 — Specialization 비용 측정

**소요 시간:** 4시간 = 4 × (40분 학습 + 20분 휴식)
**방식:** 실습 + 분석
**선수:** [4-3](4-3.md) 완료 (대상 응용 선택)

---

## 학습 목표

- 4-3 의 선택 응용에 맞춰 NPUConfig 를 **과감하게 튜닝**
- 튜닝 NPU 의 *대상 응용 metric* 개선 측정
- *다른 응용 회귀* 측정 → **specialization 비용** 정량화
- 4-5 발표를 위한 *문제 → 구조 → 결과* 스토리

> 좋은 specialization 은 *공짜가 아니다*. 다른 워크로드에서 무엇을 잃었나? 가 핵심.

## Block 1 — 응용별 튜닝 전략 (40분)

### 자율주행 → Latency 줄이기

| 변경 | 효과 | 위험 |
| --- | --- | --- |
| Clock ↑ (3 → 4 GHz) | latency ↓ | 전력 ↑ |
| PE array ↓ (32×32 → 16×16) | underutil 워크로드 latency ↓ | 큰 워크로드 throughput ↓ |
| Dataflow 변경 (WS → OS) | partial sum reuse, batch=1 친화 | weight 메모리 압박 |

### 챗봇 → Throughput per $ 늘리기

| 변경 | 효과 | 위험 |
| --- | --- | --- |
| SRAM ↑ (16 → 64 MB) | weight reuse → DRAM ↓ | 면적 ↑ |
| INT4 도입 | lane 밀도 2× | 정확도 ↓ |
| PE array ↑ (16 → 32) | 큰 batch util ↑ | 작은 batch underutil |

### 영상처리 → TOPS/W 늘리기

| 변경 | 효과 | 위험 |
| --- | --- | --- |
| INT8 또는 INT4 | MAC 에너지 ↓ | 정확도 ↓ |
| Clock ↓ | 전력 ↓ | latency ↑ |
| PE array 축소 | static power ↓ | 큰 워크로드 underutil |

> 위 표는 *시작점*. *Defining feature 와 충돌하지 않게* 튜닝.

## Block 2 — 튜닝 적용 + 측정 (40분)

### 학생 과제 (바이브 코딩)

> 1. 4-3 선택 응용의 우선 metric 정의
> 2. base NPUConfig 복제 → tuned config (위 표 또는 본인 아이디어)
> 3. base 와 tuned 둘 다로 *3개 응용 모두* 실행

### 결과 표

| 응용 | 우선 metric | base | tuned | 변화 |
| --- | --- | --- | --- | --- |
| 자율주행 | latency_ms | | | |
| LLM 챗봇 | tokens/sec/$ | | | |
| 영상 | TOPS/W | | | |

대상 응용 행은 굵게.

### 검증 기준

- 대상 응용에서 우선 metric 개선 *측정 가능* (>10%)
- 적어도 1 다른 응용에서 회귀 *측정 가능* (>5%)
- 회귀가 0 이면 *공짜 점심* — 의심 (cost 모델 누락?)

### 의도된 결과 패턴

- 대상 개선 < 다른 회귀: specialization 미미
- 대상 개선 >> 다른 회귀: specialization 가치 있음
- 본인 결과 *판단* + 보고서에 기록

## Block 3 — Specialization 비용 + 발표 자료 (40분)

### Specialization Efficiency

```
specialization_efficiency = (대상 metric 개선 %) / (다른 응용 회귀 %의 합)
```

- ≥ 2.0: 효율적
- 1.0 ~ 2.0: 적당
- < 1.0: 더 잃었음 — 재고 필요

### 발표 자료 초안 (5 슬라이드, markdown)

| 슬라이드 | 내용 |
| --- | --- |
| 1. 문제 정의 | 어느 응용? 왜? 우선 metric? |
| 2. 출발점 | Defining feature (3-1) + base 측정값 |
| 3. 튜닝 결정 | 변경 파라미터 + *왜* + 다른 변경 안 한 이유 |
| 4. 결과 | 대상 개선 % + 다른 회귀 % + spec_eff |
| 5. 한 줄 평 + 다음 단계 | *"이 NPU 는 ___ 에 ___ 만큼 빠르고, 대가로 ___ 에서 ___ 만큼 잃었다."* |

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 모든 응용 개선만 됨 | 변경이 단순 ↑ (SRAM 무한, cost 무시) | 4-2 cost 모델 적용, 면적/전력 측정 |
| 대상이 거의 개선 안 됨 | 변경이 진짜 병목과 무관 | 4-3 약점 분석으로 돌아가기 |
| spec_efficiency 음수 | 개선 < 0 | 변경 방향 반대 |
| Tuning 너무 보수적 | 안전한 작은 변경만 | *defining feature 더 강하게 밀기* |
| 비교 어려움 | metric 단위 응용마다 다름 | normalized 변화율 (%) 로 |
