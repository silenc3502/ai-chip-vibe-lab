# 4-4. 응용 맞춤 튜닝 — Specialization 비용까지 측정하기

**소요 시간:** 4시간
**방식:** 실습 + 분석 (실습 70%, 분석 30%)
**선수:** [4-3](4-3.md) 완료, 대상 응용 선택 완료

---

## 학습 목표

- 4-3에서 선택한 응용에 맞춰 NPUConfig를 **과감하게 튜닝**한다.
- 튜닝 NPU의 *대상 응용 metric* 개선을 측정한다.
- *다른 응용에서의 회귀* 를 측정해 **specialization 비용** 을 정량화한다.
- 4-5 발표를 위한 *문제 → 구조 → 결과* 스토리를 정리한다.

## 4-3과의 연결

4-3: *어느* 응용에 본인 NPU가 잘 맞는지, *어디서* 부족한지 식별.
4-4: 그 응용에 *맞춰서 다시 만들기*. 단 *비용*도 같이 측정.

> 좋은 specialization은 *공짜가 아니다*. 다른 워크로드에서 무엇을 잃었는가? 가 핵심 질문.

## Part A. 응용별 튜닝 전략 (45분)

### 자율주행 → Latency 줄이기

| 변경 | 효과 | 위험 |
| --- | --- | --- |
| Clock ↑ (3GHz → 4GHz) | latency 직선 감소 | 전력 ↑ |
| PE array ↓ (32×32 → 16×16) | underutilized 워크로드의 latency 감소 | 큰 워크로드에서 throughput ↓ |
| Dataflow 변경 (WS → OS) | partial sum reuse, batch=1 친화 | weight 메모리 압박 |
| KV 또는 weight 사전 로딩 | 첫 query latency ↓ | 메모리 압박 |

### 챗봇 → Throughput per $ 늘리기

| 변경 | 효과 | 위험 |
| --- | --- | --- |
| SRAM ↑ (16MB → 64MB) | weight reuse → DRAM 부담 ↓ | 면적 ↑ ($↑) |
| INT4 도입 | lane 밀도 2배 → throughput 2배 | 정확도 ↓ (양자화 어려움) |
| PE array ↑ (16×16 → 32×32) | 큰 batch에서 utilization ↑ | 작은 batch underutilized |
| Decode 가속 unit (small GEMV) | decode 단계 빨라짐 | 면적 ↑ |

### 영상처리 (엣지) → TOPS/W 늘리기

| 변경 | 효과 | 위험 |
| --- | --- | --- |
| INT8 또는 INT4 | MAC 에너지 ↓ → TOPS/W ↑ | 정확도 ↓ |
| Clock ↓ (1.5GHz → 1GHz) | 전력 ↓ | latency ↑ |
| PE array 축소 | static power ↓ | 큰 워크로드 underutilized |
| SRAM 축소 | 면적 ↓ → static power ↓ | DRAM 의존 ↑ → 동적 power ↑ (trade-off) |

> 위 표는 *시작점*. 학생이 *defining feature와 충돌하지 않게* 튜닝.

## Part B. 튜닝 적용 + 측정 (90분)

### 학생 과제 (바이브 코딩으로)

> 1. 4-3에서 선택한 응용의 *우선 metric*을 정의.
> 2. base NPUConfig를 복제, *tuned config*를 만들고 위 표 또는 본인 아이디어로 변경 적용.
> 3. base와 tuned 두 config로 *3개 응용 모두* 실행 (대상 응용 + 다른 2개).
>
> 결과 표:
>
> | 응용 | 우선 metric | base | tuned | 변화 |
> | --- | --- | --- | --- | --- |
> | 자율주행 | latency_ms | | | |
> | LLM 챗봇 | tokens/sec/$ | | | |
> | 영상 | TOPS/W | | | |
>
> *대상 응용 행*에 음영 또는 굵게.

### 검증 기준

- 대상 응용에서 우선 metric 개선이 *측정 가능* (>10% 이상)
- 다른 응용 중 적어도 1개에서 *회귀가 측정 가능* (>5% 이상)
- 회귀가 0이면 변경이 *공짜 점심* — 의심해 보기 (가능하지만 드뭄)

### 의도된 결과

- *대상 metric 개선 < 다른 응용 회귀* 면 specialization 효과 미미
- *대상 metric 개선 >> 다른 응용 회귀* 면 specialization 가치 있음
- 학생은 본인 결과가 어느 쪽인지 *판단*하고 보고서에 기록

## Part C. Specialization 비용 분석 + 4-5 자료 정리 (75분)

### Specialization 비용 정량화

다음 metric으로 환원 (`tuning_report.md`에 표):

```
specialization_efficiency =
    (대상 응용 metric 개선 %) / (다른 응용 metric 회귀 %의 합)
```

- > 2.0: 효율적 specialization
- 1.0 ~ 2.0: 적당
- < 1.0: 더 잃었음 — 튜닝 재고 필요

### 4-5 발표 자료 초안

다음 슬라이드 5장을 준비 (markdown으로 작성, 발표 시 그대로 표시 OK):

#### Slide 1. 문제 정의 (1장)
- 어느 응용? 왜 이 응용?
- 그 응용의 우선 metric

#### Slide 2. 본인 NPU의 출발점 (1장)
- Defining feature (3-1에서)
- base config의 metric 측정값

#### Slide 3. 튜닝 결정 (1장)
- 변경한 파라미터
- *왜* 그 변경인가 (1줄)
- 다른 변경을 안 한 이유

#### Slide 4. 결과 — 측정 (1장)
- 대상 응용 개선 %
- 다른 응용 회귀 %
- specialization_efficiency

#### Slide 5. 한 줄 평 + 다음 단계 (1장)
- "이 NPU는 ___ 응용에 ___ 만큼 빠르고, 그 대가로 ___ 응용에서 ___ 만큼 잃었다."
- 더 시간이 있다면 무엇을 시도할 것인가?

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 모든 응용에서 개선만 됨 | 변경이 단순 ↑ (예: SRAM 무한대) — 비용 추정 누락 | 4-2의 cost 모델 적용, 면적/전력도 측정 |
| 대상 응용이 거의 개선 안 됨 | 변경이 응용의 진짜 병목과 무관 | 4-3 약점 분석으로 돌아가서 *진짜 병목*에 맞는 변경 |
| Specialization efficiency가 음수 | 개선 < 0 | 변경 방향 반대 (예: 줄여야 하는데 늘림) |
| Tuning이 너무 보수적 | "안전한" 작은 변경만 함 | *defining feature를 더 강하게 밀기*, 과감한 변경 권장 |
| 비교가 어려움 | metric 단위가 응용마다 다름 | 표는 그대로 두고 *normalized 변화율 (%)* 로 비교 |
