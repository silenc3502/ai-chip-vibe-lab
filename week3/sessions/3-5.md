## Part A. 병목 분석 이론 (30분)

### Roofline 기반 진단

| 위치 | 진단 | 흔한 원인 | 흔한 처방 |
| --- | --- | --- | --- |
| 점이 BW 사선 위 (memory-bound) | DRAM/SRAM에 발목 | reuse 부족 | 더 큰 SRAM, 더 좋은 dataflow, 더 작은 dtype |
| 점이 peak 수평선 위 (compute-bound, peak 근접) | PE에 발목 | utilization 좋음, 더 PE가 필요 | PE array 키우기, 정밀도 낮추기 |
| 점이 *둘 다 아래* (underutilized) | tile/PE 모양 미스매치 | 워크로드 차원이 PE 차원과 안 맞음 | tiling 변경, PE 모양 변경 |

### Utilization 기반 진단

- pe_utilization < 0.5 → 거의 항상 underutilization 문제
- pe_utilization > 0.8 + memory-bound → 좋은 utilization인데 BW가 발목

### Energy 기반 진단

- DRAM 비율 > 50% (총 에너지 중) → DRAM 줄이기 (SRAM 키우기, reuse 늘리기)
- MAC 비율 > 80% → 효율적. 더 줄이려면 dtype을 낮춰야

| 항목 | 값 | 해석 |
| --- | --- | --- |
| pe_utilization | | |
| AI (FLOPs/byte) | | |
| AI vs critical AI (peak/BW) | | |
| DRAM 에너지 비율 | | |
| 진단 (memory / compute / underutilized) | | |

#### 2. 병목의 *근본 원인* 가설 (1~2 문장)

> 예: *"W2(LLM 단일 추론)에서 underutilized. 이유는 K=4096이 PE array의 K 차원(16)으로 256번 tile되는데, batch=1이라 input reuse가 없어서 매 tile마다 weight 재로드. SRAM이 작아서 weight tile이 reuse되지 않고 매번 DRAM에서 가져옴."*

#### 3. 1회 설계 변경 제안

다음 중 *하나*를 골라 변경:

| 옵션 | 효과 |
| --- | --- |
| SRAM 크기 ↑ (예: 4MB → 32MB) | DRAM read 횟수 감소 |
| Dataflow 변경 (WS → OS) | 다른 reuse 패턴 |
| Dtype 더 작게 (FP16 → INT8) | 같은 BW로 더 많은 element |
| PE array 모양 변경 (16×16 → 32×8) | 비대칭 워크로드 적합 |
| Multi-array (16×16 → 16×16 ×4) | 큰 batch에 유리 |

> *제약*: 다이 면적이 크게 늘면 안 됨. 대략 SRAM 8배 ≈ PE array 4배 ≈ 다이 면적 1.5~2배. 현실적 범위 내에서.

## Part C. 변경 적용 + 재측정 (90분)

### 학생 과제 (바이브 코딩으로)

> NPUConfig를 복제해 변경된 v2를 만든다. 같은 5 워크로드를 두 NPU(v1, v2)로 모두 실행. 다음 비교 표 출력:
>
> | Workload | v1 latency | v2 latency | 변화율 | v1 util | v2 util | 변화 |
>
> 각 워크로드에 대해 변화가 *어디서 왔는지* 한 줄.

### 검증 기준

- *목표* 워크로드(Part B에서 고른 약한 워크로드)에서 latency가 줄어들거나 utilization이 올라감
- 적어도 1개의 *다른* 워크로드에서 변화 (개선이든 악화든) — 변경이 trade-off 없으면 의심
- 변경의 *근본 원인 가설*이 측정으로 *증명* 또는 *반증*

### 의도된 결과 패턴

- **개선 + trade-off**: 목표 워크로드 개선, 다른 워크로드 악화. 제일 흔하고 정상.
- **개선 + free lunch**: 모든 워크로드가 같이 개선. 가능하지만 드문 — 의심해 보기.
- **개선 안 됨**: 가설이 틀림. *왜 틀렸나* 분석이 학습 포인트.

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 변경 후 모든 워크로드가 똑같음 | NPUConfig를 변경했지만 시뮬레이터가 config 값을 안 씀 | 시뮬레이터 코드에서 config 필드 사용 위치 확인 |
| 개선이 너무 큼 (10x 이상) | 변경 *전*에 어떤 cost를 카운트 안 했음 | 진단 표를 다시, *어떤 cost가 갑자기 사라졌는지* |
| Trade-off 없음 | 변경이 단순 ↑ (예: SRAM 무한대) — 비현실적 | 다이 면적 제약 고려, 다른 workload에서 회귀가 나오는 변경으로 |
| 가설이 측정과 반대 | 정상. *그게 학습* | 보고서에 그대로 적기, *왜 가설이 틀렸나* 분석 |
