## Block 1 — 병목 분석 + 본인 NPU 진단 (40분)

### Roofline 기반 진단

| 위치 | 진단 | 처방 |
| --- | --- | --- |
| BW 사선 위 (memory-bound) | DRAM/SRAM 발목 | SRAM ↑, dataflow 변경, dtype ↓ |
| peak 수평선 위 (compute-bound) | PE 발목 | PE array ↑, dtype ↓ |
| *둘 다 아래* (underutilized) | tile/PE 미스매치 | tiling/PE 모양 변경 |

### Utilization 기반

- `pe_util < 0.5` → underutilization 문제
- `pe_util > 0.8 + memory-bound` → 좋은 util 인데 BW 가 발목

## Block 2 — 1회 설계 변경 결정 + 적용 (40분)

### 병목의 근본 원인 가설 (1-2 문장)

> 예: *"W2 (LLM 단일 추론) 에서 underutilized. K=4096 이 PE K 차원(16) 으로 256번 tile 되는데 batch=1 이라 input reuse 없음. SRAM 작아서 weight 매 tile DRAM 재로드."*

### 변경 옵션 (하나 선택)

| 옵션 | 효과 | 비용 (다이 면적) |
| --- | --- | --- |
| SRAM 크기 ↑ | DRAM read ↓ | ~kB 당 0.0008 mm² |
| Dataflow 변경 (WS → OS) | 다른 reuse 패턴 | ~0 |
| Dtype ↓ (FP16 → INT8) | 같은 BW 더 많은 elem | PE 면적 ↓ |
| PE array 모양 변경 (16×16 → 32×8) | 비대칭 워크로드 적합 | ~0 |
| Multi-array (16×16 → 16×16 ×4) | 큰 batch 유리 | ~4× |

> 제약: 다이 면적 1.5-2× 이내. SRAM 8× ≈ PE 4× ≈ 면적 1.5-2×.

### 학생 과제 (바이브 코딩)

> NPUConfig 복제해 v2 생성. 같은 5 워크로드를 v1 / v2 로 모두 실행.

## Block 3 — 재측정 + Trade-off 분석 (40분)

### 비교 표 출력

| Workload | v1 lat | v2 lat | 변화 % | v1 util | v2 util |
| --- | --- | --- | --- | --- | --- |

각 워크로드 변화의 *원인* 한 줄.

### 의도된 결과 패턴

- **개선 + trade-off** (정상): 목표 개선, 다른 악화
- **개선 + free lunch** (드뭄): 모두 개선 — 의심 (다이 면적 안 잰 것?)
- **개선 안 됨** (학습): 가설 틀림 — *왜 틀렸나* 분석이 학습 포인트

### 검증 기준

- 목표 워크로드 개선 또는 *측정 가능한 회귀 + 분석*
- 적어도 1개 다른 워크로드 변화 (trade-off 없으면 의심)
- 가설이 측정으로 *증명 또는 반증*

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 변경 후 모든 워크로드 똑같음 | config 변경했지만 시뮬레이터가 안 씀 | config 필드 사용 위치 확인 |
| 개선이 너무 큼 (10x+) | 변경 *전* 어떤 cost 카운트 안 됨 | 진단 표 다시 — *어떤 cost 가 사라졌나* |
| Trade-off 없음 | 변경이 단순 ↑ (SRAM 무한) | 다이 면적 제약 + 회귀 나오는 변경 |
| 가설이 측정과 반대 | 정상 — *그게 학습* | 보고서에 *왜 가설이 틀렸나* 분석 |
| MAC 변경 후 RTL 검증 안 함 | 3-2 cocotb 회귀 누락 | `cd 02_mac_verilog && make` 로 통과 확인 |
