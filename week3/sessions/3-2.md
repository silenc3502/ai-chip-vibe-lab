## Block 1 — MAC 이론 + Verilog 첫 모듈 구조 시연 (40분)

### 1.1 MAC 정의 (10분)

`acc <- acc + (a × b)` 한 번이 1 MAC. 정의상 1 MAC = 2 FLOP.

### 1.2 Accumulator precision (15분)

| 입력 dtype | 곱 결과 | 권장 accumulator | 비고 |
| --- | --- | --- | --- |
| INT8 × INT8 | INT16 | **INT32** | 양자화 추론 표준 |
| INT4 × INT4 | INT8 | INT16 | LLM 양자화 |
| FP16 × FP16 | FP16 | **FP32** | mixed precision 학습 |
| BF16 × BF16 | BF16 | **FP32** | TPU 학습 |

**누적 정밀도가 입력보다 높은 게 일반적.** 이유: N 개를 더하면 오차/범위 모두 √N ~ N 배 늘어남 → 입력 정밀도로는 부족.

### 1.3 Per-MAC 영역과 에너지 (10분)

| dtype | 곱셈기 영역 (상대) | MAC 에너지 (pJ, 7nm) |
| --- | --- | --- |
| FP32 × FP32 | 1.0 | ~3.7 |
| FP16 × FP16 | 0.3 | ~1.1 |
| INT8 × INT8 | 0.1 | ~0.2 |

→ **같은 다이 면적에서 INT8 lane 은 FP32 의 ~10배.** NPU 가 저정밀에 베팅하는 진짜 이유.

### 1.4 Verilog 첫 모듈 구조 (5분, 강사 시연)

[Primer](../../templates/verilog-primer.md) §1, §2 를 짧게 복습 + `mac.v` 의 빈 골격을 칠판/IDE 에 그려보기:

```verilog
module mac (
    input  wire                clk,
    input  wire                rst,
    input  wire signed  [7:0]  in_data,
    input  wire signed  [7:0]  weight,
    output reg  signed  [31:0] acc
);
    always @(posedge clk) begin
        if (rst) acc <= 32'sd0;
        else     acc <= acc + (in_data * weight);
    end
endmodule
```

> Block 2 에서 학생이 *비슷한 구조를 본인 손으로* 작성합니다.

## Block 2 — Python `MACUnit` 클래스 작성 (40분)

### 2.1 학생 과제 (바이브 코딩으로)

> Python `dataclass` 없이 일반 클래스로 작성:
>
> ```python
> class MACUnit:
>     def __init__(self, input_dtype: str, weight_dtype: str, accum_dtype: str): ...
>     def reset(self): ...
>     def step(self, input_val, weight_val): ...
>     @property
>     def output(self): ...
>     @property
>     def per_mac_energy_pj(self) -> float: ...
> ```
>
> dtype 은 `'int8' | 'int16' | 'int32' | 'fp16' | 'bf16' | 'fp32'`. NumPy dtype 캐스팅을 활용. Energy 표는 1.3 표 사용.

### 2.2 의도된 실패 — INT8 input × INT8 accum

> **테스트 1 (실패 의도):** `MACUnit("int8", "int8", "int8")` 로 길이 1000 의 모두-1 벡터 dot product. 결과가 1000 이 *아닐* 것 (오버플로우). 결과 출력 후 *왜 틀렸는지* 한 줄로 적기.

> **테스트 2 (정답):** `MACUnit("int8", "int8", "int32")` 로 같은 dot product. 결과가 1000.

### 2.3 검증 기준

- 테스트 2: NumPy `np.dot()` 결과와 일치 (`atol=0` 정수)
- 테스트 1: 오버플로우 발생 명시적 출력

### 2.4 Lab 파일 위치

`week3/lab/02_mac.py` 에 작성. 3-3 NPU 통합 시뮬레이터가 import 해서 사용.

## Block 4 — Python ↔ Verilog 비교 + 회고 (40분)

### 4.1 Cross-check 결과 분석 (15분)

`test_cross_check` 가 통과하면:
- 같은 입력 시퀀스에 대해 RTL 과 Python 이 같은 누적값
- *추상화 레벨이 달라도 알고리즘은 같다* 는 첫 직접 증거

통과 안 하면:
- signed/unsigned 캐스팅? (Python `np.int8` 과 Verilog `signed [7:0]`)
- 클럭 한 사이클 지연 — Verilog 의 `acc` 는 *다음* clk edge 에 갱신
- `feed_pair` 가 acc 를 읽는 *시점*이 정확한지 확인

### 4.2 Surfer 으로 dataflow 시각화 (10분)

```bash
surfer dump.vcd &
```

**처음 열면 wave 영역이 비어 있음** (정상). 신호를 *명시적으로 추가* 해야 그래프 표시:

1. 좌측 *Variables panel* 에서 `mac` 모듈 클릭
2. 5개 신호 (`acc`, `clk`, `in_data`, `rst`, `weight`) 를 더블클릭 또는 우클릭 → *Add to wave*
3. `f` 키 — fit to view (전체 시뮬 시간 보이게)
4. `acc` 우클릭 → *Format → Decimal (signed)* (음수 누적 정상 표시)

매 clk edge 에서 acc 가 어떻게 누적되는지 *눈으로* 확인. 자세한 사용법은 [`templates/verilog-primer.md`](../../templates/verilog-primer.md) §8 참고.

학생이 회고에 *"내가 본 가장 인상적인 신호 변화"* 한 줄.

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| `iverilog` not found | brew install 누락 또는 PATH 문제 | `brew install icarus-verilog`, 새 터미널 |
| `cocotb` import 실패 | venv 활성화 안 됨 | `source .venv/bin/activate` |
| `make: cocotb-config: command not found` | venv 활성화 안 됨 (위와 같은 원인) | 같음 |
| INT8 dot product 결과가 NumPy 와 안 맞음 (Python) | NumPy 캐스팅 시 silent overflow | 누적을 명시적으로 `np.int32` 캐스팅 |
| Verilog `acc` 결과가 음수처럼 큰 값 | unsigned 로 해석됨 | `wire signed [7:0]`, `reg signed [31:0]` 명시 |
| RTL 과 Python 의 결과가 1 cycle 차이 | RTL 은 다음 clk edge 에 갱신 | feed_pair 후 추가 1 clk 대기 |
| `assert dut.acc.value == 1000` 실패 | `value` 는 `LogicArray`, 비교 시 변환 필요 | `dut.acc.value.to_signed()` |
| Surfer 빈 화면 | dump.vcd 가 없음 또는 시뮬 미실행 | `make` 먼저 실행, `dumpfile/dumpvars` 가 mac.v 에 있는지 |
