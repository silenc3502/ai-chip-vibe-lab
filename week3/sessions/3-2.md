## Part A. MAC 유닛 이론 (45분)

### MAC = Multiply-Accumulate

`acc += a * b` 한 번이 1 MAC. 보통 정의에서 1 MAC = 2 FLOP.

### 핵심 결정 — Accumulator Precision

| 입력 dtype | 곱 결과 dtype | accumulator dtype | 비고 |
| --- | --- | --- | --- |
| INT8 × INT8 | INT16 | **INT32** | 양자화 추론의 표준 |
| INT4 × INT4 | INT8 | INT16 | LLM 양자화 |
| FP16 × FP16 | FP16 | **FP32** | mixed precision 학습 |
| BF16 × BF16 | BF16 | **FP32** | TPU 학습 |
| FP8 × FP8 | FP8 | **FP32** 또는 FP16 | H100, B100 |

> **누적 정밀도가 입력보다 높은 게 일반적**. 이유: N개를 더하면 오차가 √N 이상 누적, 입력 정밀도로는 부족.

### MAC 유닛 = 곱셈기 + 누산기

```
input ─┐
       ├─[multiplier]─[adder]──> partial_sum
weight ─┘                  ↑
                       prev_partial_sum
```

- **FMA** (fused multiply-add): 곱셈과 덧셈을 1 instruction으로. 라운딩이 1번 → 정확도 ↑.
- 영역 (area): adder가 multiplier보다 작음 → 다이 면적의 70-80%가 곱셈기

### Per-MAC 영역과 에너지 (대략)

| dtype 조합 | 곱셈기 영역 (상대) | MAC 에너지 (pJ, 7nm 기준) |
| --- | --- | --- |
| FP32 × FP32 | 1.0 | ~3.7 |
| FP16 × FP16 | 0.3 | ~1.1 |
| INT8 × INT8 | 0.1 | ~0.2 |
| INT4 × INT4 | 0.04 | ~0.08 |

→ **같은 다이 면적에서 INT8 lane은 FP32의 ~10배.** 이것이 NPU가 저정밀에 베팅하는 진짜 이유.

## Part B. MAC 유닛 시뮬레이션 (75분)

### 학생 과제 (바이브 코딩으로)

> 다음 시그니처의 클래스를 만들어라:
>
> ```python
> class MACUnit:
>     def __init__(self, input_dtype: str, weight_dtype: str, accum_dtype: str): ...
>     def reset(self): ...                      # accumulator를 0으로
>     def step(self, input_val, weight_val): ...  # acc += input * weight, 한 사이클
>     @property
>     def output(self): ...                      # 현재 누적값
>     @property
>     def per_mac_energy_pj(self) -> float: ...  # 위 표 기반 추정값
> ```
>
> dtype은 'int8', 'int16', 'int32', 'fp16', 'bf16', 'fp32' 지원. NumPy의 dtype 캐스팅을 활용.
> 단위 테스트: 길이 100 벡터의 dot product를 MACUnit으로 누적한 결과가 NumPy의 `np.dot()`과 `np.allclose`.

### 검증 기준

- INT8 입력 × INT8 weight, INT32 누적: 길이 100 벡터 dot product가 NumPy와 일치
- FP16 입력 × FP16 weight, FP32 누적: NumPy와 일치 (atol=1e-3)
- *입력과 누적을 같은 dtype으로 두면* (예: INT8 × INT8, INT8 누적): 오버플로우 발생을 학생이 직접 관찰 (의도된 실패!)

### 의도된 실패 실험

- 길이 1000의 모두-1 벡터 × 모두-1 벡터를 INT8 누적으로 dot — 오버플로우. 결과 ≠ 1000.
- *왜 accumulator는 더 큰 dtype이어야 하는지* 가 직접 보임.

## Part C. 메모리 계층 모델링 (90분)

### 3-tier 메모리 모델

| 계층 | 크기 | 사이클당 BW | 접근 latency | 에너지/byte |
| --- | --- | --- | --- | --- |
| Register / Local (PE 내부) | 수십 byte | "무한" (PE 내부) | 0 cycle | ~0.1 pJ |
| **SRAM (on-chip scratchpad)** | KB ~ MB | 수백 GB/s | 1-3 cycle | ~5 pJ |
| **DRAM (off-chip HBM)** | GB | 수십~수천 GB/s | 100+ cycle | ~100 pJ |

### Memory 클래스 시그니처

```python
class Memory:
    def __init__(self, name: str, size_bytes: int, bw_bytes_per_cycle: float,
                 latency_cycles: int, energy_pj_per_byte: float): ...
    def read(self, n_bytes: int) -> tuple[int, float]:
        """returns (cycles, energy_pj)"""
    def write(self, n_bytes: int) -> tuple[int, float]: ...
    @property
    def total_reads(self) -> int: ...   # 누적 read bytes
    @property
    def total_writes(self) -> int: ...
    @property
    def total_energy_pj(self) -> float: ...
```

### 학생 과제 (바이브 코딩으로)

> 위 `Memory` 클래스를 구현. SRAM과 DRAM 두 인스턴스를 만들어 본인 NPUConfig 값과 일치시킨다. 간단한 `tier` 헬퍼를 둬서 *"이 데이터를 어디서 읽을지"* 결정 — capacity check, 더 작으면 SRAM에서, 아니면 DRAM에서.

### 단순화 가정 (이번 회차에서)

- 캐시 동작(replacement, prefetch) 모델링 안 함 — 데이터 위치는 학생/스케줄러가 명시
- BW를 사이클당 byte로 단순화 (실제는 channel/bank 등 복잡)
- 동시 접근 / contention 무시

> 이 가정들은 3-3에서 일부 보완. 지금은 *cost를 카운트할 수 있는* 인터페이스가 목적.

### 검증 기준

- 1 KB read → SRAM에서 처리, latency ≤ DRAM
- 100 MB read → SRAM 용량 초과, DRAM 자동 사용
- 모든 접근의 누적 에너지가 합리적 (예: 1 GB 읽기 = ~100 pJ × 10^9 = 100 mJ → 0.1 J. 맞음)

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| INT8 dot product 결과가 NumPy와 안 맞음 | NumPy 캐스팅 시 오버플로우 silently 발생 | 누적을 명시적으로 INT32로 (`np.int32`), 비교 시 같은 dtype으로 캐스팅 |
| Memory.read의 cycles가 항상 0 | latency_cycles가 0이거나 size 계산 오류 | latency + (n_bytes / bw_bytes_per_cycle)으로 |
| SRAM이 항상 DRAM보다 빠르게 안 나옴 | bw 단위 혼동 (GB/s vs bytes/cycle) | clock_ghz로 환산: bw_bytes_per_cycle = bw_GBps / clock_ghz |
| MAC 에너지가 비현실적 | dtype별 에너지 표를 안 적용 | 위 표 사용, 또는 학생이 자체 조사한 값 |
