## Block 1 — 5 워크로드 정의 + 첫 측정 (40분)

### 1.1 워크로드 셋 (Week 2-4 와 정렬)

| ID | 명칭 | (M, K, N) | dtype | 의미 |
| --- | --- | --- | --- | --- |
| W1 | Tiny MatMul | (32, 32, 32) | config 따름 | underutilization 점검 |
| W2 | LLM 단일 추론 | (1, 4096, 4096) | config 따름 | batch=1, weight 큰 |
| W3 | LLM 배치 추론 | (256, 4096, 4096) | config 따름 | weight reuse 큰 |
| W4 | 학습 step | (1024, 1024, 1024) | config 따름 | 균형, 큰 케이스 |
| W5 | Attention QK | (1024, 128, 1024) | config 따름 | K 작음, MN 큼 |

> dtype 은 본인 NPUConfig 따름. W4 가 4096³ 이면 너무 느릴 수 있어 1024³ 권장.

### 1.2 학생 과제 (바이브 코딩)

> 본인 `NPU.run(A, B)` 시뮬레이터를 5개 워크로드에 대해 실행. 각 워크로드 표 출력:
>
> | Workload | cycles | latency_ms | TOPS (measured) | DRAM read MB | DRAM write MB | energy_mJ | pe_util |

### 1.3 검증 기준

- W1 (tiny): pe_utilization 낮음 (PE array underutilized)
- W3 (배치 추론): pe_utilization 가장 높음 (~1.0)
- W4 (큰 균형): peak TOPS 의 50~80% 달성
- *항상* peak 에 가까우면 무언가 잘못됨 (DRAM access 미카운트)


`test_cycle_compare.py` 가 다음을 출력해야:

```
RTL    :  ~1025 cycles, acc = <NumPy ground truth>
Python :  ~2025 cycles (predicted), acc = <같은 값>
NumPy ground truth                  acc = <같은 값>
Cycle ratio (Python / RTL): ~2×
```

### 4.2 결과 해석 (10분)

**값은 같음, 사이클은 다름.** 무엇을 의미하는가:

- **결과값 일치** = 알고리즘은 같다. 두 추상화 레벨 (Python 모델, RTL) 이 *같은 산술* 을 한다.
- **사이클 차이** = 모델의 *가정* 이 RTL 과 다름. Python NPU 모델은 1 μs (= 1000 cycle) *dispatch overhead* 를 더함. RTL 은 reset 후 즉시 시작 → overhead 없음.
- 즉:
  - RTL pure compute = K cycles + 1-2 settle
  - Python prediction = 1000 (overhead) + ~K (compute)
  - **Ratio ~2×** for K=1024

### 4.3 토의 — 어느 게 *더 정확*?

- 단일 호출 latency 만 보면 RTL 측정값이 정답
- 시스템 전체 latency (host → NPU 명령 전달 포함) 는 Python 모델이 더 가까울 수도
- 본인 응용 (자율주행/챗봇/영상) 에 따라 어느 모델이 적합한지 판단

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 모든 워크로드가 peak TOPS 의 90%+ 도달 | DRAM access / underutilization 미카운트 | 3-3 unit test 회귀 |
| 본인 NPU 가 *모든* 워크로드 winner | 비교 칩 reuse_factor 가 낮게 설정됨 | Week 2-4 reuse_factor 검증 |

| Cycle ratio 가 매우 큼 (5x+) | overhead 모델 너무 큼 또는 K 너무 작음 | overhead 1 μs 가 default. K 늘리면 비율 감소 |
