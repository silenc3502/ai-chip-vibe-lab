## Block 1 — 시뮬레이터 아키텍처 (40분)

### 정확도 스펙트럼

| 종류 | 시간 단위 | 우리 선택 |
| --- | --- | --- |
| Analytical (roofline) | 한 번에 | Week 1-4 |
| **Trace-driven** (추상 cycle) | tile 단위 | **본 회차** |
| Cycle-accurate | 실제 cycle | (선택) |
| RTL simulation | gate level | (학교 밖, 3-2 RTL 만 일부) |

### 클래스 구조

```
NPU (NPUConfig 받음)
├── PEArray   ← 2-2 SystolicArray 일반화
├── SRAM      ← 3-2 Memory
├── DRAM      ← 3-2 Memory
└── Stats     ← cycles, energy, dram_bytes 누적
```

### `run(A, B)` 의사코드

```
1. A, B 를 DRAM 에 둠
2. tile loop (m, n, k):
    a. SRAM 으로 한 tile (BW + latency 카운트)
    b. PE array 로 tile MatMul (cycles 카운트)
    c. partial sum SRAM 에 누적
3. SRAM → DRAM (output write)
4. return RunResult(C, cycles, energy, dram_bytes, pe_util)
```

## Block 2 — NPU 클래스 통합 (40분)

### 학생 과제 (바이브 코딩)

> ```python
> @dataclass
> class RunResult:
>     C: np.ndarray
>     cycles: int
>     energy_pj: float
>     dram_read_bytes: int
>     dram_write_bytes: int
>     pe_utilization: float
> ```
>
> `NPU.__init__(config)` 로 PEArray/SRAM/DRAM 초기화. `NPU.run(A, B)` 가 위 RunResult 반환.

### Tile loop (Weight Stationary 가정)

```
for m_tile, n_tile, k_tile:
    load weight tile to PE
    stream input tile
    accumulate partial sum
write final output to DRAM
```

### Cycle 카운팅

- Tile load: `tile_bytes / bw_per_cycle + latency`
- Systolic execute: `~2N + PE_dim - 1` (2-2)
- Tile store: read 와 동일

### Utilization

- M < pe_rows 면 underutilized → < 1.0
- 사이클 가중 평균

## Block 3 — 4개 단위 테스트 (40분)

### `test_npu.py` 작성

| 테스트 | 워크로드 | 기대 |
| --- | --- | --- |
| 1. Exact fit | 16×16×16 (PE 16×16) | `pe_util == 1.0`, allclose |
| 2. Large multiple | 64×64×64 (PE 16×16) | `pe_util ≈ 1.0`, cycles ≈ 64 tile × per-tile |
| 3. Underutilized | 8×8×8 (PE 16×16) | `pe_util ≈ 0.125`, allclose |
| 4. Non-square | M=32, K=128, N=64 | allclose, cycles 합리적 |

### 검증 기준

- 4 테스트 모두 `np.allclose` 통과
- `pe_utilization` 이 직관과 일치 (작은 워크로드 → 작은 util)
- 2-2 사이클 수와 *같은 자릿수* (메모리 latency 추가로 약간 더 큼)

### 함정 가르치기

- 처음엔 *DRAM read 잘못 누적해서 cycle 폭증* — SRAM 의 의미 직접 체험
- 작은 K (=4) 의 systolic underutilization 학생이 직접 발견

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| `np.allclose` 실패 | tile 경계 partial sum 손실 | k_tile 루프 누적 보존, output 은 *모든* k_tile 끝나고 SRAM 저장 |
| cycles 너무 큼 | DRAM access 매 tile (reuse 없음) | WS 면 weight tile 을 multiple input 에 재사용 |
| `pe_utilization > 1.0` | 계산 오류 | 매 사이클 *active PE / total PE* 평균 |
| Test 4 무한 루프 | non-square tile 경계 처리 | `range(0, M, pe_rows)` + 마지막 tile 부분 util |
| 한 번에 다 짜려 함 | 통합이 너무 큼 | Test 1 부터 → 점차 확장 |
