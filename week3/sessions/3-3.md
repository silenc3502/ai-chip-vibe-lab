## 학습 목표

- 3-1 (config) + 3-2 (MAC, Memory) + 2-2 (systolic array)를 합쳐 **통합 NPU 시뮬레이터**를 만든다.

> 이 회차의 끝에 학생은 `npu = NPU(config); result = npu.run(workload)` 라는 한 줄로 *본인 NPU의 워크로드 실행*을 수행할 수 있어야 합니다.

## Part A. 시뮬레이터 아키텍처 (45분)

### 시뮬레이션 정확도 스펙트럼

| 종류 | 시간 단위 | 속도 | 정확도 | 우리 선택 |
| --- | --- | --- | --- | --- |
| Analytical (roofline) | "한 번에" | 매우 빠름 | ±2x | Week 1-4 |
| Trace-driven | 추상 사이클 | 빠름 | ±20% | **본 회차** |
| Cycle-accurate | 실제 사이클 | 느림 | ±5% | (선택 도전) |
| RTL simulation | gate level | 매우 느림 | 기준치 | (학교 밖) |

> 우리는 *trace-driven*을 채택. 한 사이클 = 한 systolic step. PE 안의 세부 latency는 무시.

### 클래스 다이어그램

```
NPU (NPUConfig를 받음)
├── PEArray   ← 2-2의 SystolicArray, 일반화
├── SRAM      ← 3-2 Memory
├── DRAM      ← 3-2 Memory (선택)
└── Stats     ← cycles, energy, dram_bytes 누적
```

### `run(workload)` 의사코드

```
def run(workload: Workload) -> RunResult:
    1. workload.A, B를 DRAM에 둔다
    2. tile loop:
        a. SRAM으로 한 tile 가져온다 (BW와 latency 모두 카운트)
        b. PE array로 tile MatMul 실행 (cycles 카운트)
        c. partial sum SRAM에 저장
    3. SRAM의 결과를 DRAM으로 write
    4. return (cycles, energy, dram_bytes, C)
```

## Part B. NPU 클래스 통합 (90분)

### 학생 과제 (바이브 코딩으로)

> 다음을 수행하는 `NPU` 클래스를 만들어라:
>
> 1. `__init__(self, config: NPUConfig)` — config로 PEArray, SRAM, DRAM, MACUnit 인스턴스 초기화
> 2. `run(self, A, B) -> RunResult`:
>    - A, B를 DRAM에 로드 (DRAM read 카운트)
>    - PE array 차원에 맞춰 tile 분할
>    - 각 tile을 SRAM으로 프리페치, systolic execute
>    - 모든 사이클 / 에너지 / 메모리 access 누적
>    - 결과 C를 NumPy로 반환
>
> ```python
> @dataclass
> class RunResult:
>     C: np.ndarray
>     cycles: int
>     energy_pj: float
>     dram_read_bytes: int
>     dram_write_bytes: int
>     pe_utilization: float   # 0.0 ~ 1.0
> ```

### 핵심 모듈

#### Tile loop

```
for m_tile in range(0, M, ARR_ROWS):
    for n_tile in range(0, N, ARR_COLS):
        for k_tile in range(0, K, ARR_COLS):  # weight stationary 가정
            load weight tile to PE array
            stream input tile through PE
            accumulate partial sum (in PE or SRAM, dataflow에 따라)
        store output tile to SRAM
write SRAM → DRAM
```

#### Cycle 카운팅

- Tile load: `tile_bytes / bw_bytes_per_cycle + latency`
- Systolic execute: `~2N + (PE 차원) - 1` 사이클 (2-2와 같음)
- Tile store: read와 동일

#### Utilization

- 한 tile에서 PE array를 부분만 채우는 경우 (예: M < ARR_ROWS) utilization < 1.0
- 평균 utilization을 누적하고 사이클 가중 평균

## Part C. 단위 테스트 + 검증 (75분)

### 학생 과제 — `test_npu.py`

다음 4가지 테스트 케이스를 작성하고 모두 통과시킨다.

#### Test 1: PE array에 정확히 맞는 케이스

- 16×16 PE array, 16×16 MatMul (M=K=N=16)
- 결과가 `A @ B` 와 `np.allclose`
- pe_utilization == 1.0 (또는 매우 근접)

#### Test 2: PE array보다 큰 케이스

- 16×16 PE array, 64×64 MatMul (M=K=N=64)
- 결과 검증
- cycles ≈ 4×4×4 = 64 tile × per-tile cycles
- pe_utilization ≈ 1.0

#### Test 3: PE array보다 작은 케이스

- 16×16 PE array, 8×8 MatMul (M=K=N=8)
- 결과 검증
- pe_utilization ≈ (8×8) / (16×16) = 0.25

#### Test 4: Non-square 워크로드

- M=32, K=128, N=64
- 결과 검증
- cycles 계산이 합리적

### 검증 기준

- 4개 테스트 모두 `np.allclose` 통과
- pe_utilization이 직관에 맞음 (작은 워크로드 → 작은 utilization)
- 2-2 시뮬레이터와 cycle 수가 *같은 자릿수* (perfect match일 필요 없음 — 추가 메모리 latency가 더해졌으니 약간 더 큼)

### 함정 가르치기

- 학생이 처음에 *DRAM read 누적을 잘못해서 cycle이 크게 나옴*. 정상 — 그래서 SRAM이 있다는 점을 직접 체험.
- 작은 K (예: K=4)에 대한 systolic underutilization을 학생이 발견하게.

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| `np.allclose` 실패 | tile 경계에서 partial sum 손실 (해당 tile의 누적이 다음 tile로 넘어가지 않음) | k_tile 루프에서 누적 보존, output tile은 *모든* k_tile 끝나고 SRAM 저장 |
| cycles가 너무 큼 | DRAM access를 매 tile마다 함 (tile reuse 없음) | weight stationary면 한 weight tile을 multiple input tile에 재사용 |
| pe_utilization > 1.0 | utilization 계산 오류 | 매 사이클의 *active PE 수* / *total PE*를 평균 |
| Test 4가 무한 루프 | non-square tile loop의 경계 처리 | `range(0, M, ARR_ROWS)` + 마지막 tile의 부분 utilization 처리 |
| 한 번에 다 짜려고 | 통합이 너무 큼 | 먼저 Test 1만 통과 → 점차 확장 |
