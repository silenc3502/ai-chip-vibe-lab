## Part B. 첫 바이브 코딩 (90분)

### 과제

> **1000 × 1000 float32 행렬 두 개를 (1) Python for-loop, (2) NumPy** 두 가지 방식으로 곱하고, 각 방식의 실행 시간을 측정·비교하는 스크립트를 만든다.

이 과제를 **직접 코드로 작성하지 말고, 자연어 지시로 에이전트에게 시켜서** 만듭니다.

### 진행 단계 (프롬프트 로그 5단계와 1:1 매칭)

1. **의도 정리** — 무엇을 왜 하려는지 한두 문장으로. (`prompt-log.md`에 먼저 적기)
2. **프롬프트 작성** — 위 과제를 자연어로 풀어서 입력.
3. **결과 확인** — 에이전트가 만든 코드를 `week1/lab/01_loop_vs_numpy.py`로 저장하고 실행.
4. **검증** — 결과 수치가 합리적인가?
   - for-loop이 NumPy보다 **수십~수백 배 느려야** 정상.
   - NumPy 결과가 GFLOPs 기준 **10~200 GFLOPs** 범위면 합리적.
   - 두 방식의 결과 행렬이 **수치적으로 같은가** (`np.allclose`)?
5. **재수정** — 빠진 게 있다면 추가 지시. 예:
   - warmup 1회 누락 → 첫 실행이 비정상적으로 느림
   - 결과 검증 코드 누락 → 두 방식 결과가 같은지 확인 안 됨
   - 1000×1000 for-loop이 너무 오래 걸림 → 200×200으로 축소 요청

### 예상 결과 (참고용)

```
Loop  : 12.4 s   (≈ 0.16 GFLOPs)
NumPy : 0.018 s  (≈ 110 GFLOPs)
Speedup: 689x
allclose: True
```
*숫자는 기기마다 다릅니다. 자릿수만 비슷하면 OK.*

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| `ModuleNotFoundError: numpy` | 가상환경 비활성 | `source .venv/bin/activate` |
| for-loop이 5분 넘게 안 끝남 | 1000×1000은 순수 Python에 너무 큼 | 200×200으로 축소 후 재측정 |
| 첫 실행만 비정상적으로 느림 | warmup 효과 (캐시/JIT) | 측정 전 1회 warmup 추가 |
| 두 결과가 미세하게 다름 | float 누적 오차 | `np.allclose` (절대오차 허용) 사용 |
| 에이전트가 GPU 코드를 만듦 | 의도 불명확 | "CPU에서만, 외부 라이브러리는 NumPy만" 명시 |
