## Block 1 — SIMD 이론 (40분)

### 핵심 개념

- **SIMD** = 한 instruction 이 여러 데이터를 동시 처리
  - 스칼라: `c = a + b` — 1 cycle, 1 float
  - SIMD: `[c0..c7] = [a0..a7] + [b0..b7]` — 1 cycle, 8 float

### Lane 폭

| ISA | 레지스터 폭 | float32 lane | 비고 |
| --- | --- | --- | --- |
| SSE | 128 bit | 4 | x86 모든 현대 CPU |
| AVX2 | 256 bit | 8 | Intel Haswell+ / AMD Zen+ |
| AVX-512 | 512 bit | 16 | 서버급 |
| NEON | 128 bit | 4 | ARM (Apple Silicon) |

### Peak GFLOPS 공식

```
peak = clock(GHz) × SIMD_lane × FMA_factor(=2 if FMA)
```

예: 3 GHz × AVX2 (8 lane) × FMA → 48 GFLOPS/core. 8 코어 = ~384 GFLOPS.

### 왜 Python 루프는 SIMD 를 못 쓰나

- 인터프리터가 element 하나씩 dispatch
- 각 element 가 PyObject — 타입 체크, refcount...
- SIMD instruction 을 emit 할 자리 없음
- **NumPy 는 C/Fortran 컴파일된 BLAS 호출** → SIMD 가 그 안에서 동작

## Block 2 — 실험 1: 이론 peak vs 측정 활용률 (40분)

### 학생 과제 (바이브 코딩)

> Python 스크립트로:
> 1. 본인 CPU 모델, 클럭, 코어 수, SIMD ISA (`py-cpuinfo` 사용)
> 2. 이론적 peak GFLOPS (단일 / 전체 코어)
> 3. NumPy N×N MatMul (N=4096) 측정 GFLOPS
> 4. 활용률 = 측정 / 이론

### 검증 기준

- 일반: 측정 GFLOPS 가 이론의 **30~70%** 범위
- **Apple Silicon 예외**: 활용률 > 100% (AMX 매트릭스 코프로세서가 BLAS 가속). 정상 — *단순 vector lane 공식은 매트릭스 엔진을 capture 못함*
- 활용률이 5% 미만 → BLAS 가 single-thread (`OPENBLAS_NUM_THREADS` 확인)

## Block 3 — 실험 2: 700배 분해 (40분)

### 세 가지 변형

벡터 길이 N=1M ~ 10M, element-wise 합:

| 변형 | 데이터 | 실행 방식 | 핵심 |
| --- | --- | --- | --- |
| (a) | Python list | `[a[i]+b[i] for i in range(N)]` | 1-1 베이스라인 |
| (b) | **NumPy 배열** | `for i: c[i] = a[i] + b[i]` (Python 루프) | **핵심 surprise** |
| (c) | NumPy 배열 | `c = a + b` (벡터화) | NumPy 본연 |

### 학생 과제 (바이브 코딩)

> 길이 1M 의 두 벡터로 (a) (b) (c) 시간 측정. 비율 출력. 셋의 결과 정합성 (`np.allclose`) 확인.

### 검증 기준

- (a) 와 (b) 자릿수 비슷 (Apple Silicon 에서 (b) 가 (a) 보다 2-3배 느릴 수 있음 — NumPy boxing overhead)
- (c) 가 (b) 보다 **수백 배** 빠름
- (b) → (c) 점프가 *SIMD + 벡터화 효과* 모두 보여줌

### 분해 식 (회고용)

```
1-1 의 ~700x 차이 ≈
    캐시 효과 (1-2)        × 약 ____ x  (2-5x)
  × SIMD lane (1-3 이론)    × 약 ____ x  (4-16x)
  × 인터프리터 오버헤드     × 약 ____ x  (실험2 (b→c) ÷ SIMD lane)
```

자릿수가 비슷하면 OK.
