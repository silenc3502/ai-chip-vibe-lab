## Part A. SIMD 이론 (45분)

### 핵심 개념

- **SIMD = 한 instruction이 여러 데이터를 동시에 처리**
  - 스칼라 add: `c = a + b` — 1 cycle, 1 float
  - SIMD add: `[c0..c7] = [a0..a7] + [b0..b7]` — 1 cycle, 8 float

- **lane 폭 (architecture별)**

  | ISA | 레지스터 폭 | float32 lane | 비고 |
  | --- | --- | --- | --- |
  | SSE | 128 bit | 4 | x86 모든 현대 CPU |
  | AVX2 | 256 bit | 8 | Intel Haswell+ / AMD Zen+ |
  | AVX-512 | 512 bit | 16 | 서버급, 일부 데스크탑 |
  | NEON | 128 bit | 4 | ARM (Apple Silicon, 모바일) |
  | SVE/SVE2 | 가변 | 4~64 | 최신 ARM 서버 |

- **FMA (Fused Multiply-Add)** — `d = a*b + c`를 1 instruction으로. peak 계산 시 ×2.

- **이론적 peak GFLOPS (single core, single precision)**

  ```
  peak = clock(GHz) × SIMD_lane × FMA_factor(=2 if FMA available)
  ```

  예: 3 GHz, AVX2 (8 lane), FMA → 3 × 8 × 2 = **48 GFLOPS / core**.
  8 코어 모두 사용하면 ~384 GFLOPS.

- **왜 Python 루프는 SIMD를 못 쓰나?**
  - 인터프리터는 element 하나씩 dispatch
  - 각 element가 PyObject (참조, 타입 체크, refcount...)
  - SIMD instruction을 emit하는 컴파일러가 끼어들 자리가 없음
  - **NumPy는 C/Fortran으로 컴파일된 BLAS 호출** → 그 안에서 SIMD가 동작

## Part B. 실험 1 — 이론적 peak vs 측정 peak (75분)

### 실험 의도

본인 CPU의 이론적 peak GFLOPS를 계산하고, NumPy MatMul로 실제 도달 가능한 throughput을 측정해서 *활용률(utilization)*을 구한다.

### 학생 과제 (바이브 코딩으로)

> Python 스크립트로 다음을 출력하라:
> 1. 본인 CPU의 모델, 클럭, 코어 수, 지원 SIMD ISA (AVX2, AVX-512, NEON 등)
> 2. 위 정보로 계산한 단일 코어 / 전체 코어 이론적 peak GFLOPS (float32)
> 3. NumPy로 N×N MatMul (N=2048 or 4096)을 실행한 측정 GFLOPS
> 4. 활용률(%) = 측정 / 이론

힌트로 줘도 되는 정보:
- macOS: `sysctl -a | grep machdep.cpu` 또는 `platform` 모듈
- Linux: `/proc/cpuinfo`
- 라이브러리: `cpuinfo` (pip install py-cpuinfo)

### 검증 기준

- 이론적 peak: M2 Mac 기준 ~200 GFLOPS/core × 8 ≈ 1.5 TFLOPS 정도. 데스크탑 x86은 보통 더 높음.
- 측정 GFLOPS는 이론의 **30~70%** 범위면 정상 (NumPy 백엔드가 OpenBLAS/MKL/Accelerate 어느 것이냐에 따라 다름).
- 활용률이 100%에 매우 가깝거나 0%에 가까우면 **계산 식이 틀렸을 가능성** — 다시 점검.

### 흔한 막힘

- SIMD ISA 식별 실패 → `py-cpuinfo` 패키지 설치 후 `cpuinfo.get_cpu_info()['flags']` 확인
- Apple Silicon에서 클럭이 표시 안 됨 → 공식 사양(예: M2 = 3.5 GHz P-core, 2.4 GHz E-core)을 코드에 상수로 두는 것도 OK
- N이 작아서 GFLOPS가 낮음 → N=4096부터 시작 (작은 N은 캐시에 들어가버려 측정 노이즈가 큼... 사실 반대로 작은 N은 *L3 fit*돼서 빠르게 나옴, 변동이 큼)

## Part C. 실험 2 — 700배 분해하기 (90분)

### 실험 의도

같은 *vector add* 연산을 세 가지 방식으로 실행하고 시간을 비교해서, **어디서 속도 차이가 오는지**를 분리한다.

### 세 가지 변형

벡터 길이 N = 10,000,000 (10M) float32 두 개의 element-wise 합:

| 변형 | 데이터 | 실행 방식 | 핵심 |
| --- | --- | --- | --- |
| (a) | Python list | `for i: c[i] = a[i] + b[i]` | 1-1과 같은 베이스라인 |
| (b) | **NumPy 배열** | `for i: c[i] = a[i] + b[i]` (Python 루프!) | **여기가 핵심 surprise** |
| (c) | NumPy 배열 | `c = a + b` (벡터화) | NumPy 본연 |

### 학생 과제 (바이브 코딩으로)

> 길이 10M의 두 벡터를 만들어서 (a) Python list + Python 루프, (b) NumPy ndarray + Python 루프, (c) NumPy ndarray + 벡터 연산 세 방식으로 element-wise 합을 계산하고 각 시간을 측정해서 비율을 출력하는 스크립트. 결과 정합성을 위해 셋의 합 결과가 모두 같은지 확인.

> ⚠️ (a)와 (b)는 매우 느릴 수 있음. N=10M이 부담되면 N=1M으로 줄여도 됨. 단 같은 N으로 셋 다 비교.

### 검증 기준

- (a)와 (b)는 **시간이 비슷**해야 한다 (둘 다 Python 인터프리터가 매 element를 dispatch).
  - (b)가 (a)보다 약간 느릴 수도 있음 — NumPy scalar가 Python float보다 indexing 비용이 큼.
- (c)는 (b)보다 **수백 배** 빠르다.
- 즉 (b) → (c)의 점프가 *"인터프리터 + SIMD"* 효과를 모두 보여줌.

### 분해 해석

학생이 회고에 채워야 할 식:

```
1-1의 ~700x 차이 ≈
    캐시 효과 (1-2)         × 약 ____ x
  × SIMD lane (1-3 이론)     × 약 ____ x  (4 ~ 16)
  × 인터프리터 오버헤드      × 약 ____ x  (실험 2의 (a→c) 비율 ÷ SIMD lane)
```

> 정확한 곱이 안 맞아도 **자릿수가 비슷**(수백 ~ 수천 배)하면 OK. 학생이 "왜" 차이가 나는지 *프레임*을 갖는 게 목표.
