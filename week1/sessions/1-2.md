## Part A. 메모리 계층 이론 (60분)

### 강의 핵심 개념

- **레이턴시 차이 (대략)**

  | 단계 | 크기 | 접근 시간 | 비고 |
  | --- | --- | --- | --- |
  | Register | ~32개 × 8B | < 1 ns | CPU 사이클당 |
  | L1 cache | 32~64 KB | ~1 ns | 코어별 |
  | L2 cache | 256 KB ~ 1 MB | ~3-5 ns | |
  | L3 cache | 8~64 MB | ~10-30 ns | 코어 간 공유 |
  | DRAM | ~수 GB | ~80-150 ns | 메인 메모리 |

  → DRAM 한 번 vs L1 한 번이 **100배** 차이.

- **캐시 라인** — 캐시는 64 byte 단위로 가져옴. float32 16개가 한 라인.
- **공간 지역성 (spatial locality)** — 인접한 데이터를 곧 쓸 가능성이 높다는 가정.
- **시간 지역성 (temporal locality)** — 최근 쓴 데이터를 곧 또 쓸 가능성이 높다는 가정.
- **C-contiguous (row-major)** — NumPy 기본. 한 행이 메모리에 연속.

### 토의 질문 (실습 전 5분)

## Part B. 실험 1 — Working set 크기별 접근 시간 (90분)

### 실험 의도

배열 크기 N을 1 KB부터 256 MB까지 늘려가며, 동일한 연산(예: 전체 합)에 걸리는 *element당* 시간을 측정. 캐시 계층 경계에서 **계단식 증가**가 보여야 한다.

### 학생 과제 (바이브 코딩으로)

다음을 자연어로 에이전트에게 시켜서 만들어라.

> 배열 크기 N을 약 10단계로(예: 1KB, 4KB, 16KB, ..., 64MB, 256MB) 변화시키며 각 크기에서 NumPy 배열의 element 평균 접근 시간(나노초/element)을 측정하는 스크립트. 결과를 표 또는 matplotlib 그래프로 출력. 측정 안정성을 위해 warmup과 반복 평균을 포함할 것.

### 검증 기준

- 결과 그래프가 X축 log scale일 때 **2~3개의 step**(L1, L2, L3 → DRAM 경계)을 보여야 한다.
- 가장 작은 크기(L1 안)와 가장 큰 크기(DRAM)의 element당 시간 비율이 **수십~100배** 차이 나야 한다.
- 절대값은 기기마다 다르지만 자릿수는 위 표(이론)와 비슷해야 한다.

### 결과가 안 나올 때 점검

- 측정 시간이 너무 짧으면 노이즈가 큼 → 반복 횟수를 늘려 총 측정시간 ≥ 100 ms 확보
- step이 안 보이면 → 크기 단계가 너무 성김. L1 경계(32-64 KB) 주변을 더 촘촘히
- 모든 크기가 비슷하게 빠름 → 컴파일러/NumPy가 공통 부분식을 최적화했을 수 있음. 매 반복마다 다른 데이터를 쓰도록 유도

## Part C. 실험 2 — Row-major vs Column-major 접근 (75분)

### 실험 의도

같은 NumPy 행렬(예: 4096×4096 float32, 64 MB)에 대해 행 방향 합 (`axis=1`, 연속 접근)과 열 방향 합 (`axis=0`, strided 접근)의 시간을 비교한다. 같은 연산량이지만 캐시 효율 차이로 시간이 다르게 나와야 한다.

### 학생 과제 (바이브 코딩으로)

> NumPy로 4096x4096 float32 C-contiguous 행렬을 만들고, `np.sum(A, axis=1)`(행 방향)과 `np.sum(A, axis=0)`(열 방향)의 실행 시간을 각각 10회 평균으로 측정해서 비율을 출력하는 스크립트. 결과 검증을 위해 두 합의 합계가 같은지(`np.allclose`)도 확인.

추가 도전 (시간 남으면):
- 같은 실험을 `np.asfortranarray(A)`로 만든 F-contiguous 배열에서 반복 — `axis=0`이 빨라지고 `axis=1`이 느려져야 함
- 행렬 크기를 N=512, 1024, 2048, 4096, 8192로 바꿔가며 비율이 어떻게 변하는지 (작은 N은 L3에 들어가서 차이가 작고, 큰 N은 DRAM 경합으로 차이가 큼)

### 검증 기준

- 4096×4096에서 col-wise가 row-wise보다 **2배 이상** 느려야 한다 (대개 2~5배).
- 두 합의 결과는 부동소수 오차 범위 내에서 같아야 한다.
- 행렬 크기를 키우면 비율이 일반적으로 **커진다** (캐시 압박 증가).

### 1-1 질문에 대한 답

> "왜 for-loop이 느렸나?"
>
> 1-1의 Python list-of-float는 **객체 포인터의 배열**이라 사실상 매 element 접근이 무작위 메모리 접근(=대부분 캐시 미스)이었다. NumPy 배열은 연속된 raw float이라 캐시 친화적이다. 이번 실험으로 **같은 NumPy 안에서도** 접근 패턴만으로 수 배 차이가 나는 걸 확인했다 — Python의 메모리 모델은 그보다 훨씬 가혹하다.

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 큰 배열에서 `MemoryError` | 256 MB가 RAM 부담 | 최대 크기를 64 MB로 축소 |
| Working set 그래프가 모든 크기에서 *평탄* (예: 0.1 ns/elem) | NumPy의 `sum()` 이 SIMD reduction으로 메모리 BW를 이미 포화 → cache step이 안 보임 (특히 Apple Silicon + Accelerate) | **Pointer chasing**으로 대체. 무작위 permutation을 만들고 `i = perm[i]`를 반복 — 데이터 의존성이 prefetcher를 무력화. cache 위계가 latency 차이로 명확히 보임 (참조 솔루션 02 참고) |
| col-wise가 더 *빠름* (axis=0 < axis=1) | F-contiguous 가능성 *또는* Apple Accelerate / 최신 OpenBLAS의 streaming reduction (작은 column-output 버퍼가 L1에 fit하면 strided 가정이 깨짐) | `A.flags['C_CONTIGUOUS']` 확인. Apple Silicon에서는 정상 — *plat에 따라 결과가 다를 수 있음* 을 학습 자료로 활용 |
| 비율이 1.0에 가까움 | 행렬이 너무 작아 L3에 다 들어감 | N을 4096 이상으로. 그래도 BLAS가 영리하면 차이가 작을 수 있음 |
