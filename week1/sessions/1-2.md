## Block 1 — 메모리 계층 이론 (40분)

### 레이턴시 계층 (대략)

| 단계 | 크기 | 접근 시간 |
| --- | --- | --- |
| Register | ~32 × 8B | < 1 ns |
| L1 cache | 32~64 KB | ~1 ns |
| L2 cache | 256 KB ~ 1 MB | ~3-5 ns |
| L3 cache | 8~64 MB | ~10-30 ns |
| DRAM | ~수 GB | ~80-150 ns |

→ DRAM 한 번 vs L1 한 번이 **100배** 차이.

### 핵심 개념

- **캐시 라인** — 캐시는 64 byte 단위. float32 16개가 한 라인.
- **공간 지역성** — 인접 데이터를 곧 쓸 가능성
- **시간 지역성** — 최근 쓴 데이터를 곧 또 쓸 가능성
- **C-contiguous (row-major)** — NumPy 기본. 한 행이 메모리에 연속

## Block 2 — 실험 1: Pointer chasing 으로 캐시 위계 측정 (40분)

### 실험 의도

배열 크기 N 을 1KB → 256MB 로 늘려가며 *element 당* 접근 시간 측정. 캐시 경계에서 step 이 보여야 한다.

> **NumPy `sum()` 같은 SIMD 친화적 연산은 메모리 BW 를 포화시켜 step 이 안 보일 수 있음.** Pointer chasing (data 의존성으로 prefetcher 무력화) 이 cross-platform 으로 step 을 잘 보여준다.

### 학생 과제 (바이브 코딩)

> 크기 N 의 무작위 permutation 배열을 만들고 `i = perm[i]` 를 반복하는 pointer chasing 으로 평균 접근 latency (ns/step) 측정. N 을 1KB ~ 256MB 까지 10단계로 변화. matplotlib 으로 그래프 출력.

### 검증 기준

- L1 / L2 / L3 / DRAM 경계에서 **step 변화** 보임
- 가장 작은 (L1 안) vs 가장 큰 (DRAM) 의 비율 **수십 ~ 100배**
- 절대값은 기기마다 다르지만 *경향* 은 일치

## Block 3 — 실험 2: Row vs Column 접근 (40분)

### 실험 의도

같은 4096×4096 float32 행렬에서 행 방향 합 (`axis=1`, contiguous) vs 열 방향 합 (`axis=0`, strided) 비교.

### 학생 과제 (바이브 코딩)

> NumPy 4096×4096 float32 C-contiguous 행렬에서 `np.sum(A, axis=1)` 과 `np.sum(A, axis=0)` 시간을 10회 평균으로 측정. 비율 출력. `np.allclose` 로 두 합의 합계 일치 검증.

### 검증 기준

- 일반적: col-wise (axis=0) 가 row-wise 보다 **2배 이상 느림**
- *Apple Silicon 예외*: Accelerate 의 streaming reduction 으로 col-wise 가 *더 빠를* 수 있음 — 이 경우 *플랫폼 의존성* 자체가 학습 포인트

### 1-1 질문에 대한 답

> *"왜 for-loop 이 느렸나?"*
>
> Python list of float 은 객체 포인터의 배열 → 매 element 접근이 사실상 무작위 메모리 (대부분 cache miss). NumPy 배열은 연속된 raw float 이라 cache 친화적. **같은 NumPy 안에서도** 접근 패턴만으로 수 배 차이 — Python 메모리 모델은 더 가혹.

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 큰 배열에서 `MemoryError` | 256 MB 가 RAM 부담 | 최대 크기를 64 MB 로 축소 |
| Working set 그래프 평탄 (모든 크기 비슷) | NumPy `sum()` 이 SIMD 포화로 cache step 안 보임 | **Pointer chasing** 로 대체 (참조 솔루션 02 참고) |
| col-wise (axis=0) 가 더 *빠름* | F-contiguous 가능성 *또는* Apple Accelerate streaming reduction | `A.flags['C_CONTIGUOUS']` 확인. Apple 에서는 정상 — 플랫폼 의존성 학습 자료 |
| 비율이 1.0 근처 | 행렬이 작아 L3 에 다 들어감 | N=4096 이상 |
