## Block 1 — Roofline 이론 (40분)

### 핵심 식

```
T_compute = total_flops / peak_gflops_per_sec
T_memory  = bytes_moved / peak_bandwidth_per_sec
T_actual  ≈ max(T_compute, T_memory)
```

### Arithmetic Intensity (AI)

```
AI = FLOPs / bytes_moved   (FLOPs/byte)
```

- AI 낮으면 → memory-bound
- AI 높으면 → compute-bound
- 임계 AI = peak_gflops / peak_bw_GBps

### MatMul 의 AI

N×N×N MatMul (float32):
- FLOPs = 2N³
- bytes (naive) = 12N²
- AI = N/6

N=4096 → AI ≈ 683 → **대부분 기기에서 compute-bound**

### 기기별 임계 AI

| 기기 | peak GFLOPS (FP32) | BW (GB/s) | 임계 AI |
| --- | --- | --- | --- |
| Intel i9 | ~1500 | 90 | ~17 |
| Apple M (CPU+AMX) | ~3000 | 200 | ~15 |
| RTX 4090 | ~82,000 | 1008 | ~81 |
| H100 (FP32) | ~67,000 | 3350 | ~20 |

### 토의 (5분)

2. N=64 작은 MatMul 은 어느 쪽 bound?

## Block 2 — 실험 1: CPU 모델로 측정값 검증 (40분)

### 학생 과제 (바이브 코딩)

> 다음 시그니처:
>
> ```python
> def roofline_estimate(
>     n: int, m: int, k: int,
>     peak_gflops: float,
>     peak_bw_gbps: float,
>     reuse_factor: float = 1.0,
>     dtype_bytes: int = 4,
> ) -> dict:
>     """returns {time_s, gflops, ai, bound: 'compute'|'memory'}"""
> ```
>
> 1-3 에서 알아낸 본인 CPU 의 peak 로 N=2048, 4096 예측. 1-1/1-3 측정값과 비교.

### 검증 기준

- 모델 예측이 측정값의 **0.5x ~ 2x 범위**
- N=2048, 4096 둘 다 **compute-bound** (NumPy MatMul AI 가 충분히 높음)
- **Apple Silicon 주의**: 1-3 의 *측정 peak* (예: 3000 GFLOPS) 사용. py-cpuinfo 의 단순 공식 결과 (예: 200 GFLOPS) 쓰면 5배 차이 — 이는 AMX 미반영

## Block 3 — 실험 2: GPU 파라미터로 같은 모델 (40분)

### 학생 과제 (바이브 코딩)

다음 표 출력:

| Hardware | peak GFLOPS | BW (GB/s) | N=2048 ms | N=4096 ms | bound | speedup vs CPU |
| --- | --- | --- | --- | --- | --- | --- |
| 본인 CPU | ... | ... | ... | ... | compute | 1.0x |
| RTX 4090 | 82000 | 1008 | ... | ... | ... | ... |
| H100 (FP32) | 67000 | 3350 | ... | ... | ... | ... |

### 검증 기준

- 모든 GPU 행이 **compute-bound** (큰 MatMul)
- speedup 이 *peak GFLOPS 비율과 자릿수 비슷*
- memory-bound 로 나오면 코드/입력 버그

### 함정

- *peak GFLOPS 단위 혼동* — TFLOPS 를 GFLOPS 로 입력 (RTX 4090 = 82,000 GFLOPS = 82 TFLOPS)
- *모든 게 memory-bound* — reuse_factor 누락 (실제 BLAS 는 수십 reuse)

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| GPU 예측이 CPU 보다 *느림* | TFLOPS 를 GFLOPS 로 입력 | 단위 검증 (RTX 4090 = 82,000 GFLOPS) |
| 모든 케이스가 memory-bound | reuse_factor 누락 | reuse_factor 도입, 실제 BLAS 는 ~수십 |
| CPU 모델이 측정값보다 100x 빠르게 예측 | peak 100% 활용 가정 | 활용률 0.5-0.7 곱하거나 "이론치" 명시 |
| Apple CPU 예측이 측정과 5x 차이 | py-cpuinfo 의 vector lane 공식이 AMX 미반영 | 1-3 의 *측정 peak* 사용 (보통 3000 GFLOPS for M-series) |
| AI 단위 혼동 | FLOPs/byte vs FLOPs/element | byte 기준 통일, dtype_bytes 포함 |
