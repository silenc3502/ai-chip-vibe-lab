## 학습 목표

- **Roofline 모델** — 한 워크로드가 *compute-bound*인지 *memory-bound*인지 판별하는 기본 도구를 이해한다.
- **파라미터화된 성능 모델**을 직접 vibe-code해서, CPU 스펙과 GPU 스펙을 같은 식에 넣고 예측 시간을 비교한다.
- 1-3의 *"GPU는 lane이 많다"* 라는 정성적 진술을 **수치로 정량화**한다.
- Week 3 NPU 시뮬레이터의 모델링 스타일에 미리 노출된다.

## 왜 모델인가? (실제 GPU를 쓰지 않는 이유)

1. **환경 독립성** — CUDA / Metal / ROCm 의존성 없이 모든 학생이 동일하게 실습.
2. **파라미터 변경의 자유** — "lane 수가 2배라면?", "BW가 절반이라면?" 을 *기기를 바꾸지 않고* 검증.
3. **블랙박스 감소** — 실제 GPU 라이브러리는 내부 동작을 가린다. 모델은 학생이 작성하므로 *왜* 빠른지가 식 안에 명시되어 있음.
4. **Week 3 준비** — 같은 모델링 스타일로 NPU를 만들 것임.

> 💡 GPU에 접근 가능한 학생은 **Part C 마지막의 선택 도전**에서 모델 예측과 실측을 비교해 보세요.

## Part A. Roofline 이론 (45분)

### 핵심 식

워크로드의 실행 시간은 두 가지 한계 중 *느린 쪽*이 결정한다.

```
T_compute = total_flops / peak_gflops_per_sec
T_memory  = bytes_moved / peak_bandwidth_per_sec
T_actual  ≈ max(T_compute, T_memory)
```

### Arithmetic Intensity (AI)

```
AI = FLOPs / bytes_moved   (단위: FLOPs/byte)
```

- AI가 **낮으면** memory-bound (메모리가 못 따라감)
- AI가 **높으면** compute-bound (연산기가 못 따라감)

기기마다 *임계 AI*가 있음: `AI_critical = peak_gflops / peak_bw_GBps`. 워크로드 AI가 이보다 낮으면 메모리, 높으면 연산이 병목.

### MatMul의 AI

N × K × K × M 행렬곱:

- FLOPs ≈ `2 × N × M × K`
- 순진하게 한 번만 데이터를 읽고 결과를 쓰는 경우 bytes ≈ `(N·K + K·M + N·M) × dtype_bytes`
- 정사각 N×N×N MatMul (float32):
  - FLOPs = 2N³
  - bytes (naive) = 3N² × 4 = 12N²
  - AI = 2N³ / 12N² = **N / 6**
  - N=4096 → AI ≈ 683 FLOPs/byte → **compute-bound on most hardware**

### 기기별 임계 AI 예시

| 기기 | peak GFLOPS (FP32) | BW (GB/s) | 임계 AI |
| --- | --- | --- | --- |
| Intel i9-13900K | ~1500 | 90 | ~17 |
| Apple M2 (CPU) | ~600 | 100 | ~6 |
| NVIDIA RTX 4090 | ~82,000 | 1008 | ~81 |
| NVIDIA H100 SXM | ~67,000 (FP32) | 3350 | ~20 |
| Apple M2 (GPU) | ~3,600 | 100 | ~36 |

> 숫자는 *명목 spec*이며 실 구현은 60~80%만 도달. 학생은 본인이 사용할 GPU의 spec을 직접 찾아오게 하세요.

## Part B. 실험 1 — CPU Roofline 모델로 1-1 결과 검증 (75분)

### 실험 의도

**모델을 만든 뒤, 이미 알고 있는 측정값(1-1 또는 1-3의 NumPy MatMul)으로 *모델을 검증*한다.** 모델이 CPU에서 합리적이어야 GPU 예측을 신뢰할 수 있다.

### 학생 과제 (바이브 코딩으로)

> 다음 시그니처의 함수를 만들어라:
>
> ```python
> def roofline_estimate(
>     n: int, m: int, k: int,
>     peak_gflops: float,
>     peak_bw_gbps: float,
>     reuse_factor: float = 1.0,    # 1.0 = naive, 클수록 데이터 재사용 많음
>     dtype_bytes: int = 4,
> ) -> dict:
>     """returns {time_s, gflops, ai, bound: 'compute'|'memory', utilization}"""
> ```
>
> 그리고 1-3에서 알아낸 본인 CPU의 peak_gflops, peak_bw_gbps를 넣어 N=2048, 4096 정사각 MatMul의 예측 시간과 GFLOPS를 출력. 1-1 또는 1-3에서 측정한 실제 NumPy MatMul 시간과 비교.

### 검증 기준

- 모델 예측이 측정값의 **0.5x ~ 2x 범위** 안에 들어와야 한다 (단순 모델치고 OK).
- N=2048, 4096 둘 다 **compute-bound**로 분류돼야 한다 (NumPy MatMul은 AI가 충분히 높음).
- 활용률 (measured / model_peak)이 합리적인 값 (30~80%) — model이 peak를 그대로 주는 게 아니라 효율 계수를 곱하는 식이면 더 정확해짐.

### 함정

- `reuse_factor=1.0`은 **순진한 가정**임. 실제 BLAS는 타일링으로 데이터를 수십~수백 회 재사용 → memory bound가 아니라 compute bound가 되는 이유.
- 학생이 모델에 reuse_factor를 도입했다면 1-3 measured와 더 잘 맞을 것.

## Part C. 실험 2 — GPU 파라미터로 같은 모델 돌리기 (90분)

### 실험 의도

같은 함수에 GPU의 spec을 넣어 예측 시간을 구하고, CPU 대비 speedup을 산출. *모델을 안 바꾸고 파라미터만 바꿨다*는 점이 핵심.

### 학생 과제 (바이브 코딩으로)

다음을 표로 출력하는 스크립트를 만들어라:

| Hardware | peak GFLOPS | BW (GB/s) | N=2048 예측시간 | N=4096 예측시간 | bound | speedup vs CPU |
| --- | --- | --- | --- | --- | --- | --- |
| 본인 CPU | ... | ... | ... | ... | ... | 1.0x |
| RTX 4090 | 82000 | 1008 | ... | ... | ... | ... |
| H100 SXM | 67000 | 3350 | ... | ... | ... | ... |
| Apple M2 GPU | 3600 | 100 | ... | ... | ... | ... |

### 검증 기준

- 모든 GPU 행이 **compute-bound**로 분류돼야 한다 (큰 MatMul이므로).
- speedup vs CPU가 **GPU peak GFLOPS 비율과 자릿수가 비슷**해야 한다 (model이 compute-bound 영역이므로 BW는 영향 없음).
- 만약 어떤 GPU가 memory-bound로 나온다면 코드 또는 입력에 버그.

### 추가 분석 (시간 남으면)

다음을 스윕해서 그래프 생성:

1. **N 스윕** — N을 64 ~ 8192로 변화. AI 변화에 따라 bound가 어떻게 바뀌는가?
2. **reuse_factor 스윕** — 0.1 ~ 10. 작은 reuse에서 GPU가 memory-bound로 바뀌는 지점 관찰.

### 선택 도전 (GPU 접근 가능한 학생)

- CuPy / PyTorch / MLX / MPS 중 하나로 N=4096 MatMul 실측
- 모델 예측과 비교, 활용률 계산
- *예측이 측정의 1.5x ~ 3x 범위면 모델 잘 잡은 것*

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| GPU 예측이 CPU보다 *느림* | peak GFLOPS 단위 혼동 (TFLOPS를 GFLOPS로 입력) | 단위 검증, RTX 4090 = 82,000 GFLOPS = 82 TFLOPS |
| 모든 케이스가 memory-bound | reuse_factor가 너무 낮거나 누락 | 기본을 1.0으로 두되, 실제 BLAS는 ~수십 reuse라는 점 강조 |
| CPU 모델이 측정값보다 100x 빠르게 예측 | peak를 100% 활용한다고 가정 | 활용률 0.5~0.7을 곱하거나, "이론치"임을 명시 |
| dtype_bytes를 float64=8로 두는 학생 | NumPy 기본은 float64, MatMul은 float32 권장 | 1-1에서 float32로 측정했음을 상기 |
| AI 단위 헷갈림 | FLOPs/byte vs FLOPs/element | byte 기준으로 통일, dtype_bytes를 식에 포함 |
