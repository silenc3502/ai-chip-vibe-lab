## Part A. TPU 역사와 설계 동기 (45분)

### 왜 Google이 TPU를 만들었나

- 2013: Google 음성검색에 DNN 도입 → 추론 트래픽 폭증 예상 → "데이터센터 전체를 2배로 늘려야 한다"는 추산
- 결론: 추론에 특화된 칩을 자체 설계하는 게 cheaper
- 2015: TPU v1 배포 (15개월 만에). INT8, 65,536개 MAC unit 256×256 systolic array

### TPU 세대별 변화

| 버전 | 출시 | 정밀도 | systolic | HBM | 용도 |
| --- | --- | --- | --- | --- | --- |
| v1 | 2015 | INT8 | 256×256 | (DDR3) | 추론 |
| v2 | 2017 | BF16 | 128×128 ×2 | HBM | 학습 |
| v3 | 2018 | BF16 | 128×128 ×2 | HBM | 학습 (4× v2) |
| v4 | 2022 | BF16 | 128×128 ×4 | HBM | OCS optical interconnect |
| v5 | 2023 | BF16 / INT8 | (확장) | HBM | 추론/학습 분기 |

### 설계 헌신 (architectural commitment)

- "MatMul이 99% 워크로드" 라는 가정에 *모든 다이 면적* 을 베팅
- 컨볼루션, 어텐션 등도 결국 MatMul로 환원
- 그 외 op은 호스트 CPU 또는 별도 unit이 처리

## Part B. Systolic Array 메커니즘 (75분)

### 한 PE의 동작

각 PE(Processing Element)는 다음 3개를 갖는다:

```
weight register : 가중치 한 개 (고정)
multiplier      : 입력 × 가중치
adder           : 위에서 내려온 partial sum + multiplier 결과
```

매 사이클마다:

1. 왼쪽에서 *입력 활성*이 들어옴 (한 사이클 후 오른쪽으로 흘려보냄)
2. 위에서 *partial sum*이 내려옴
3. PE가 입력 × weight를 계산하여 partial sum에 더함
4. 결과를 아래 PE에 전달

### 4×4 systolic array MatMul 추적 (강사 시연)

`C = A · B`, A는 4×4, B는 4×4 → C는 4×4.

설정 (Weight Stationary):
- B의 16개 element를 4×4 PE에 1개씩 미리 로드 (PE[i,j].weight = B[i,j])
- A의 행을 *왼쪽 가장자리*로 `staggered`하게 입력 (A의 i번째 행이 i 사이클 지연되어 들어감)
- C의 결과는 *아래쪽 가장자리*에서 나옴

8 사이클 후 모든 출력이 완료. (자세한 trace는 시뮬레이터로 확인.)

### 왜 빠른가 — Reuse 횟수

| 대상 | reuse 횟수 (4×4 systolic, 4×4 MatMul) |
| --- | --- |
| Weight (B) | 4번 (각 가중치가 A의 4개 행에 곱해짐) |
| Input (A) | 4번 (각 입력이 B의 4개 열과 곱해짐) |
| Partial sum | 한 번도 메모리에 안 나감 (PE 내부에서 누적) |

→ 데이터 한 개가 **reuse N번** 됨. naive ordering 대비 N배 적은 메모리 접근.
→ 256×256 array (TPU v1)이면 **256×256 = 65,536 MAC을 매 사이클 동시 실행**, 데이터는 모든 가장자리에서만 들어가고 나옴.

## Part C. 실험 — 4×4 Systolic Array 시뮬레이터 (90분)

### 실험 의도

사이클 단위로 PE의 상태를 추적하면서 4×4 MatMul을 수행하는 시뮬레이터를 작성. 결과가 NumPy MatMul과 일치하는지 검증.

### 학생 과제 (바이브 코딩으로)

> 4×4 systolic array를 시뮬레이션하는 Python 클래스를 만들어라. Weight Stationary 모드. 각 PE는 weight, partial_sum 두 상태를 갖는다. `step()` 메서드를 호출할 때마다 한 사이클 진행. 매 step마다 PE 격자의 상태(weight, current input, partial_sum)를 출력. 4×4 A, 4×4 B로 C = A·B를 계산하고 NumPy 결과와 `np.allclose`로 비교.

### 시뮬레이터 사양

- 클래스 `SystolicArray(rows, cols)`: 4×4 격자
- 메서드:
  - `load_weights(B)` — B 행렬을 PE에 로드
  - `feed_input(A)` — A의 행을 staggered하게 큐잉
  - `step()` — 한 사이클 진행, 모든 PE의 partial_sum 업데이트
  - `collect_output()` — 아래쪽에서 나오는 C 행 수집
- 출력: 매 사이클마다 4×4 격자의 partial_sum 표

### 검증 기준

- 시뮬레이터 결과와 `A @ B`가 `np.allclose`
- 총 사이클 수 ≈ N + N + (N-1) ≈ 2N + N-1 = 3N - 1 (4×4면 11 사이클 정도; staggering 방식에 따라 약간 다름)
- 사이클별 PE 상태 표가 출력되어 디버깅 가능

### 추가 도전 (시간 남으면)

1. 8×8 또는 N×N으로 일반화
2. K 차원 ≠ N (직사각형 MatMul) 지원
3. INT8 정밀도로 변경 (weight, input을 int8 캐스팅)

### Week 3 RTL 미리보기 (Verilog snippet)

본 회차에서 작성한 Python systolic 의 단일 PE 동작은 다음 Verilog 와 *정확히 같습니다*:

```verilog
// 3-2 에서 학생이 작성할 단일 MAC (Verilog snippet)
module mac (
    input  wire                clk,
    input  wire                rst,
    input  wire signed  [7:0]  in_data,    // weight stationary 의 input
    input  wire signed  [7:0]  weight,     // PE 에 stationary 한 weight
    output reg  signed  [31:0] acc         // partial sum 누적
);
    always @(posedge clk) begin
        if (rst) acc <= 32'sd0;
        else     acc <= acc + (in_data * weight);   // ← Python systolic 의 1 step
    end
endmodule
```

위 Verilog 의 동작은 본 회차 Python systolic 의 한 PE 와 1-to-1 대응. 차이는 *언어*가 아니라 *대상*:

- **Python**: 분석용. 디버깅 쉬움, 사이클 추적 가능, 수치 ground truth.
- **Verilog**: 구현용. 합성 가능, 실제 칩이 됨, FPGA 로 동작 가능.

3-2 에서 **본인이 직접 작성** 후 **cocotb 로 Python ↔ Verilog 결과 일치를 검증** 합니다. *같은 알고리즘의 두 추상화 레벨* — 이 회차 Python systolic 의 의미가 Week 3 에서 *실제 hardware* 로 이어집니다.

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 결과가 NumPy와 다름 | input staggering 또는 partial_sum propagation 타이밍 오류 | step별로 PE 상태를 print하며 한 사이클씩 검증 |
| 사이클 수가 예상보다 많음 | flush 단계(마지막 partial sum이 PE에서 다 빠져나오는 데 걸리는 시간) 누락 | 정상. flush 포함하면 ~3N-1 사이클 |
| 4×4 코드가 복잡해서 이해 안 됨 | 한 번에 다 짜려고 함 | 먼저 1×1 PE → 2×2 → 4×4 순으로 키우기 |
| weight stationary와 input stationary가 헷갈림 | 2-1 개념 정리 부족 | 2-1 표 다시 보기, "내가 짜는 코드에서 무엇이 PE에 *고정*되는가" 자문 |
