## Part A. MatMul의 3가지 분해와 6가지 ordering (45분)

### 세 가지 분해

`C[i,j] = sum_k A[i,k] * B[k,j]` 를 어느 차원을 outer loop에 두느냐에 따라:

- **Inner product** (k가 inner) — 한 출력 element를 끝까지 누적
  ```
  for i: for j: for k: C[i,j] += A[i,k] * B[k,j]
  ```
- **Outer product** (k가 outer) — 한 K-slice의 outer product를 매 단계 더함
  ```
  for k: for i: for j: C[i,j] += A[i,k] * B[k,j]
  ```
- **Gustavson** (i가 outer, k가 middle) — 한 행씩 만들면서 K로 누적
  ```
  for i: for k: for j: C[i,j] += A[i,k] * B[k,j]
  ```

### 6가지 loop ordering

`{i, j, k}`의 순열 6개. 각 ordering의 **inner-most loop**이 무엇을 접근하는가가 캐시 친화성을 결정.

| Order | 가장 안쪽 접근 | C-contiguous (row-major) 친화성 |
| --- | --- | --- |
| ijk | A[i,k], B[k,j] | A는 row 순회(친화), B는 col 순회(strided) |
| ikj | A[i,k], B[k,j], C[i,j] | A 고정, B와 C 모두 row 순회(친화) ✓ |
| jik | A[i,k], B[k,j] | A는 row 순회, B는 col 순회 (ijk와 비슷) |
| jki | A[i,k], B[k,j], C[i,j] | A와 C가 col 순회(strided) |
| kij | A[i,k], B[k,j], C[i,j] | A 고정, B와 C 모두 row 순회 ✓ |
| kji | A[i,k], B[k,j], C[i,j] | A와 C가 col 순회(strided) |

> 직관: **`ikj`와 `kij`가 가장 빠를 것** (가장 안쪽이 row 순회). 학생이 실험에서 확인.

## Part B. 실험 — 6가지 loop ordering 시간 비교 (90분)

### 실험 의도

Pure Python(또는 NumPy 인덱싱)으로 6가지 ordering MatMul을 구현하고 시간을 측정. 가장 빠른 / 가장 느린 것이 직관과 일치하는지 확인.

### 학생 과제 (바이브 코딩으로)

> N=200 정사각 float32 NumPy 배열 두 개 A, B에 대해 C = A @ B를 6가지 loop ordering(ijk, ikj, jik, jki, kij, kji)으로 구현한 함수 6개를 만들어라. 모두 NumPy ndarray 인덱싱(`C[i,j] += A[i,k] * B[k,j]`)을 쓰되, NumPy 벡터 연산은 *쓰지 말 것*. 각 ordering의 시간을 측정하고 표로 출력. 결과 검증을 위해 모두 `np.allclose(C, A @ B)`로 비교.

> ⚠️ N=200이 너무 느리면 N=100으로 축소. NumPy `@`는 ~1ms로 끝나지만 6 ordering의 Python 루프는 1~10초 수준.

### 검증 기준

- 6 결과 모두 `np.allclose(..., A @ B)` True.
- 가장 빠른/느린 ordering의 시간 비율: 일반적으로 **1.5~5배** 이지만, Apple Silicon + 최신 NumPy처럼 *element 접근이 매우 효율적*인 환경에서는 **1.05x 미만**일 수 있음 (인터프리터 오버헤드가 캐시 차이를 압도).
- 학습 목표는 *개념 이해* — `ikj`, `kij`가 *왜* 빠를 후보인지를 설명할 수 있는가가 측정 수치보다 중요.
- 명확한 차이를 보고 싶다면 **Numba `@njit`** 으로 컴파일 (`pip install numba`, 선택). 이 경우 5~20배까지 벌어짐.

### 함정

- NumPy 인덱싱은 매 access가 PyObject 변환을 거쳐서 **인터프리터 오버헤드가 메모리 효과보다 큼**. 그래도 차이는 측정 가능.
- 더 극적인 결과를 보고 싶으면 Numba `@njit`으로 같은 6 함수를 컴파일해서 비교 (선택 도전).

## Part C. Dataflow 개념 — Stationary 분류 (75분)

### "Stationary" 정의

> Inner-most loop에서 **인덱스가 변하지 않는** 변수.

예시:

- `for i: for j: for k: C[i,j] += A[i,k] * B[k,j]` (ijk)
  - inner-most는 k. C[i,j]는 k에 의존하지 않음 → **C가 stationary**
  - 이런 ordering을 **Output Stationary** 라고 부름.
- `for k: for i: for j: ...` (kij)
  - inner-most는 j. A[i,k]는 j에 의존하지 않음 → **A가 stationary**
  - **Input Stationary** (A가 입력 활성화일 때).
- `for j: for k: for i: ...` (jki)
  - inner-most는 i. B[k,j]는 i에 의존하지 않음 → **B가 stationary**
  - **Weight Stationary** (B가 가중치일 때).

### 6 ordering의 분류

| Order | inner-most | stationary | 분류 |
| --- | --- | --- | --- |
| ijk | k | C | **Output Stationary** |
| jik | k | C | **Output Stationary** |
| ikj | j | A | **Input Stationary** |
| kij | j | A | **Input Stationary** |
| jki | i | B | **Weight Stationary** |
| kji | i | B | **Weight Stationary** |

### Dataflow 다이어그램 그리기 (`dataflow_diagrams.md`에 포함)

각 stationary에 대해 다음 4 요소를 그림으로 표현:

1. PE(또는 register)에 *고정*된 데이터 (= stationary)
2. *흘러 들어오는* 데이터 (= moving)
3. *흘러 나가는* 데이터 (= output)
4. 한 사이클의 동작 (multiply-accumulate)

> 손그림을 사진으로 찍어 `figures/`에 저장해도 OK. 핵심은 *어느 화살표가 안쪽으로 들어오고 어느 화살표가 바깥으로 나가는가*.

### 핵심 통찰

- **6 ordering = 3 stationary × 2 (다른 차원의 외부 순서)**.
- 하드웨어는 이 중 **하나를 골라서 실리콘에 못박는** 것 — 2-2 (TPU)에서 weight stationary가 어떻게 실리콘이 되는지 봅니다.
- "어느 stationary가 좋은가?" 는 워크로드에 따라 다름 — 2-3에서 측정.

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 6 ordering의 시간 차이가 거의 없음 (1.05x 미만) | Python 인터프리터 오버헤드(~30ns/access)가 cache miss 비용(~5ns)을 압도 | 정상. *학습 목표는 측정 수치가 아니라 개념*. Numba `@njit` 컴파일하면 5-20x 차이 (선택 도전) |

| stationary 분류가 헷갈림 | "inner-most에서 변하지 않는 변수"의 정의를 놓침 | 표 다시 보기, 자기 코드의 inner-most loop 변수를 표시 |
