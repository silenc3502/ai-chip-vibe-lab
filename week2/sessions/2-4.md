## Part A. TPU를 Roofline 모델에 추가 (60분)

### 1-4 모델 복습

```python
def roofline_estimate(n, m, k, peak_gflops, peak_bw_gbps, reuse_factor=1.0, dtype_bytes=4):
    flops = 2 * n * m * k
    bytes_naive = (n*k + k*m + n*m) * dtype_bytes
    bytes_actual = bytes_naive / reuse_factor
    t_compute = flops / (peak_gflops * 1e9)
    t_memory = bytes_actual / (peak_bw_gbps * 1e9)
    t_actual = max(t_compute, t_memory)
    ...
```

### TPU 파라미터 매핑

TPU v4 spec (대략):

| 파라미터 | TPU v4 |
| --- | --- |
| peak (BF16) | 275 TFLOPS |
| peak (INT8) | 275 TOPS (BF16과 같음, MXU는 동일 array) |
| HBM BW | ~1.2 TB/s |
| systolic array | 128×128 ×4 chips |
| 효율 reuse | array dimension × tiling 효과 |

**reuse_factor 추정**:

- 128×128 systolic on N×N MatMul (N ≥ 128): 한 데이터가 ~128번 reuse → reuse_factor ≈ 128
- 작은 MatMul (N < 128): 어레이가 부분만 사용됨 → reuse < 128
- 매우 큰 MatMul (N >> 128): outer tile loop이 추가 reuse를 만듦 → reuse > 128

### 같은 모델에 *다양한 칩* 넣기

| 칩 | dtype | peak | BW | reuse_factor |
| --- | --- | --- | --- | --- |
| 본인 CPU (NumPy/BLAS) | fp32 | 측정값 | 측정값 | 8~16 (vector + cache tiling) |
| RTX 4090 (FP32) | fp32 | 82,000 GFLOPS | 1008 GB/s | 32~64 (tensor core, shared mem) |
| H100 (Tensor Core, FP16) | fp16 | 989,000 GFLOPS | 3350 GB/s | 64~128 |
| TPU v4 | bf16 | 275,000 GFLOPS | 1200 GB/s | 64~128 |
| Apple Neural Engine | int8 | 15,800 GOPS | 100 GB/s | 16~32 |

> 위 reuse_factor는 *모델의 입력값*. 학생이 백서 등으로 더 정교한 추정을 해보면 좋음.

## Part B. 워크로드별 비교 (90분)

### 비교 워크로드 셋

| 워크로드 ID | (M, K, N) | dtype | 특성 |
| --- | --- | --- | --- |
| W1: Tiny MatMul | (32, 32, 32) | fp32 | 모든 array가 underutilized |
| W2: 단일 추론 | (1, 4096, 4096) | fp16 | weight 큰데 batch 1 → 재사용 작음 |
| W3: 배치 추론 | (256, 4096, 4096) | fp16 | weight 재사용 256배 |
| W4: 큰 학습 | (4096, 4096, 4096) | bf16 | 균형, 모든 array가 잘 채움 |
| W5: LLM 어텐션 (toy) | (1024, 128, 1024) | fp16 | K 작음, MN 큼 |

### 학생 과제 (바이브 코딩으로)

> 1-5에서 만든 roofline 모델을 받아서, 위 5개 워크로드 × 5개 칩 (총 25 케이스) 의 *예측 시간*과 *bound type*을 표로 출력하는 스크립트. 각 워크로드에서 가장 빠른 칩을 마킹. 결과를 matplotlib stacked bar 또는 heatmap으로 시각화.

### 검증 기준

- W1 (tiny): GPU/TPU 모두 underutilized — 의외로 **CPU가 winner** (kernel launch overhead가 결정적)
- W2 (단일 추론, 작은 batch): H100/TPU 우위 작아짐 (weight reuse가 batch 1에서 죽음). 보통 H100이 1위, TPU 2위
- W3 (배치 추론): H100/TPU 모두 큰 폭 우위. **요즘은 H100의 raw peak (989 TFLOPS FP16)이 TPU v4 (275 TFLOPS BF16)보다 높아 H100이 winner**, TPU 2위
- W4 (큰 균형 MatMul): H100 압도적 1위, TPU 2위
- W5 (LLM attention, K 작음): K=128로 systolic array 차원 부분만 사용 → 모든 칩의 우위가 W4 대비 줄어듦. 여전히 H100이 1위

> 주의: 2022년 TPU v4 발표 시점에는 W3에서 TPU가 1위였으나, 2023년 H100 등장 이후 raw peak에서 역전. *시점에 따라 winner가 바뀜* 도 학습 포인트.

### 함정 / 함정 가르치기

- *peak TFLOPS만 보고 비교하면 W1, W2의 결과를 설명할 수 없음*. reuse_factor가 워크로드 크기에 따라 변하는 걸 모델에 반영해야 함.
- TPU를 *모든 워크로드의 정답* 으로 보면 W2 결과(단일 추론 inefficient)에서 막힘.

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| TPU가 W2에서도 압도적 1위 | reuse_factor를 워크로드 크기와 무관한 상수로 둠 | reuse_factor를 *batch와 N에 의존*하게 모델링 (선형/대수 함수) |
| 모든 GPU/TPU가 memory-bound | dtype별 byte 환산 누락, 또는 reuse_factor가 너무 낮음 | 1-4의 함정 표 다시 보기 |
| W1에서 CPU가 더 빠르게 안 나옴 | tiny MatMul의 GPU/TPU underutilization을 모델에 안 반영 | reuse_factor에 `min(N, array_dim) / array_dim` 같은 효율 페널티 |
| 표 채우기는 했는데 *한 줄 요약*이 막연 | 측정값에서 인사이트 추출 훈련 부족 | "이 결과의 *원인*은 어느 파라미터 차이인가?" 자문 |
