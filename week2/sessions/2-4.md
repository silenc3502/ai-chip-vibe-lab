## Block 1 — TPU 를 Roofline 모델에 추가 (40분)

### TPU v4 spec 매핑

| 파라미터 | TPU v4 |
| --- | --- |
| peak (BF16) | 275 TFLOPS |
| HBM BW | 1.2 TB/s |
| systolic | 128×128 ×4 |
| reuse_factor | array dim × tiling |

**reuse_factor 모델링** (워크로드 크기에 따라 변함):
- N ≥ 128: reuse ≈ 128
- N < 128: reuse < 128 (어레이 부분만 사용)
- N >> 128: reuse > 128 (outer tile 추가 reuse)

### 같은 모델에 5 칩 입력

| 칩 | dtype | peak | BW (GB/s) | reuse_factor |
| --- | --- | --- | --- | --- |
| CPU (M+AMX) | fp32 | 3000 GFLOPS | 200 | 16 |
| RTX 4090 | fp32 | 82,000 | 1008 | 64 |
| H100 (Tensor) | fp16 | 989,000 | 3350 | 128 |
| TPU v4 | bf16 | 275,000 | 1200 | 128 |
| Apple NE | int8 | 15,800 | 100 | 32 |

## Block 2 — 5 워크로드 × 5 칩 표 (40분)

### 워크로드 셋

| ID | (M, K, N) | dtype | 특성 |
| --- | --- | --- | --- |
| W1 Tiny | (32, 32, 32) | fp32 | 모두 underutilized |
| W2 LLM 단일 | (1, 4096, 4096) | fp16 | weight 큰데 batch=1 |
| W3 LLM 배치 | (256, 4096, 4096) | fp16 | weight reuse 큼 |
| W4 큰 균형 | (4096, 4096, 4096) | bf16 | 모든 array 채움 |
| W5 Attn (K small) | (1024, 128, 1024) | fp16 | systolic 부분만 사용 |

### 학생 과제 (바이브 코딩)

> 1-5 의 roofline 모델 받아 5 워크로드 × 5 칩 (25 케이스) *예측 시간* + *bound type* 표 출력. 각 워크로드 winner 마킹. matplotlib heatmap 또는 stacked bar 시각화.

### 검증 기준 (2024 시점)

- W1 (tiny): GPU/TPU 모두 underutilized — **CPU winner** (kernel launch overhead)
- W2 (단일 추론): H100 1위, TPU 2위 (batch=1 이라 weight reuse 죽음)
- W3 (배치 추론): H100 winner (raw peak 989 TFLOPS), TPU 2위
- W4 (큰 균형): H100 압도적 1위
- W5 (K=128 작음): 모든 칩 우위가 W4 대비 줄어듦

> 주의: 2022 TPU v4 발표 시점엔 W3 에서 TPU 1위. 2023 H100 이후 역전. *시점에 따라 winner 바뀜* 도 학습 포인트.

> *한 줄* 좋은 예: *"많은 lane × FMA 로 throughput 높지만, control flow + 큰 cache 때문에 효율(TOPS/W) 은 NPU 보다 낮다"* (GPU)

### 토의 (15분)

1. W1 에서 CPU 가 의외로 경쟁력 있는 이유?
2. W3 에서 H100 이 가장 빠른 이유는 *peak* 때문인가 *reuse* 때문인가?
3. 워크로드를 알면 칩을 고를 수 있다 — *워크로드를 칩에 맞추는* 것은 누구의 일?

### 함정

- *peak TFLOPS 만 보고 비교* 하면 W1, W2 설명 못 함 — reuse_factor 가 워크로드 크기에 의존해야
- TPU 를 *모든 워크로드 정답* 으로 보면 W2 (단일 추론) 에서 막힘
