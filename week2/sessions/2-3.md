## Part A. Eyeriss 분류와 Trade-offs (45분)

### 4가지 dataflow (Eyeriss, MIT, 2016)

| 이름 | 무엇이 PE에 고정? | 무엇이 흐름? | 잘 맞는 워크로드 |
| --- | --- | --- | --- |
| **Weight Stationary (WS)** | weight | input, partial sum | weight reuse 큰 — 추론, 큰 batch |
| **Output Stationary (OS)** | output (partial sum 누적) | weight, input | output reuse 큰 — 작은 channel, large feature map |
| **Input Stationary (IS)** | input activation | weight, partial sum | input reuse 큰 — 큰 channel, 작은 feature map |
| **Row Stationary (RS)** | 한 행씩 | (복합) | Conv 일반화에 강함 |

> 본 회차에서는 RS는 개념만 언급하고, **WS / OS / IS 3종을 직접 시뮬레이션**.

### 메모리 접근 비용 모델 (단순화)

각 데이터의 위치는 다음 3계층 중 하나:

1. **DRAM (off-chip)** — 가장 비쌈 (~100 pJ/byte)
2. **Global SRAM (on-chip buffer)** — 중간 (~5 pJ/byte)
3. **PE register / Local** — 가장 쌈 (~1 pJ/byte)

이번 실험에서 카운트할 것:

- DRAM read: input, weight, output을 처음 가져오거나 다시 가져올 때
- DRAM write: 최종 output 저장
- Inter-PE move: PE 간 데이터 이동 (한 hop = 1)

## Part B. 실험 1 — 3가지 Stationary 메모리 접근 카운팅 (90분)

### 실험 의도

같은 N×N×N MatMul을 3가지 stationary로 시뮬해서 DRAM read/write, inter-PE move를 카운트. 어느 것이 적은가?

### 학생 과제 (바이브 코딩으로)

> 2-2에서 만든 SystolicArray를 확장 또는 새 파일에 작성. `simulate(A, B, dataflow)` 함수가 dataflow ∈ {'WS', 'OS', 'IS'} 중 하나를 받아 시뮬레이션 후 다음을 dict로 반환:
> ```python
> {
>     'cycles': int,
>     'dram_reads': int,    # bytes 단위
>     'dram_writes': int,
>     'pe_moves': int,
>     'C': np.ndarray,
> }
> ```
> 4×4 PE array에서 N=8 MatMul (즉 8×8 MatMul을 4×4 PE로 tiling)을 3 dataflow로 실행. 결과 비교.

### 단순화 가정

- Weight, Input, Output 모두 float32 (4 byte/element)
- DRAM에서 읽을 때마다 카운트 (캐싱 없다고 가정 — 시뮬 단순화)
- PE 간 한 hop 이동도 카운트

### 검증 기준

- 3가지 모두 결과 C가 `np.allclose(A @ B)`
- DRAM access 카운트가 3 dataflow에서 *서로 다름*
- WS는 weight 재사용이 큼 → weight DRAM read가 가장 적음
- OS는 output 재사용 → output DRAM write가 가장 적음
- IS는 input 재사용 → input DRAM read가 가장 적음

### 워크로드 변형 (시간 남으면)

다음 워크로드에 대해 3가지 dataflow의 총 DRAM bytes를 비교:

| 워크로드 | (M, K, N) | 특성 |
| --- | --- | --- |
| Batched FC | (1024, 512, 64) | weight 재사용 큼 |
| 단발 inference | (1, 512, 1024) | 재사용 작음 |
| 큰 sequence MatMul | (2048, 2048, 2048) | 균형 |

## Part C. 시각화 — 3종 dataflow를 한 그림에 (75분)

### 시각화 의도

3가지 dataflow에서 *어떤 데이터가 정지하고 어떤 데이터가 흐르는가* 를 한 눈에 보이게.

### 학생 과제 (바이브 코딩으로)

> matplotlib 또는 graphviz로 4×4 PE 격자 3개를 한 그림에 나란히 그려라. 각 격자 위에:
> - 정지(stationary)한 데이터를 PE 안에 사각형으로
> - 흐르는 데이터를 화살표로 (입력은 가장자리에서 들어옴, 출력은 가장자리에서 나감)
> - 제목에 dataflow 이름과 그 dataflow의 *특징 한 줄*

### 검증 기준

- 한 PNG에 3개 dataflow가 한 줄로 나열
- 각 그림에서 *고정된 데이터* 와 *흐르는 데이터* 가 시각적으로 구분 가능

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 3 dataflow의 DRAM 카운트가 비슷 | tiling이 너무 작아서 reuse 효과가 안 보임 | N=8 또는 16으로 키우고 PE array는 4×4 유지 |
| OS의 output DRAM write가 0 | 시뮬 종료 시 PE의 partial sum을 DRAM에 쓰는 단계 누락 | flush 단계 추가 |
| IS 시뮬이 OS와 거의 같음 | input과 output stationary를 코드에서 구분 안 함 | "PE에 *어떤 데이터가 register에 머무는가*" 를 명시적으로 코드에 표시 |
| 시각화가 3개 다 비슷해 보임 | stationary 데이터를 시각적으로 강조 안 함 | 색깔 또는 박스 두께로 *고정된 것을 다르게* 표시 |
