## Block 1 — Eyeriss 분류 + Trade-offs (40분)

### 4 가지 dataflow (Eyeriss, MIT 2016)

| 이름 | PE 에 고정 | 흐름 | 잘 맞는 워크로드 |
| --- | --- | --- | --- |
| **WS** | weight | input, partial sum | 추론, 큰 batch |
| **OS** | output (누적) | weight, input | 작은 channel, large feature map |
| **IS** | input | weight, partial sum | 큰 channel, 작은 feature map |
| **RS** | 한 행씩 | (복합) | Conv 일반화 |

> 본 회차는 **WS / OS / IS 3종 직접 시뮬**. RS 는 개념만.

### 메모리 접근 비용 (단순화)

| 계층 | 비용 |
| --- | --- |
| DRAM (off-chip) | ~100 pJ/byte |
| Global SRAM | ~5 pJ/byte |
| PE register | ~1 pJ/byte |

이번 실험에서 카운트:
- DRAM read: input/weight/output 처음 가져오거나 다시
- DRAM write: 최종 output
- Inter-PE move: 한 hop = 1

## Block 2 — 실험: 3 Stationary 메모리 접근 카운팅 (40분)

### 학생 과제 (바이브 코딩)

> 2-2 의 SystolicArray 확장. `simulate(A, B, dataflow)` 함수가 `'WS' | 'OS' | 'IS'` 받아 시뮬 후 다음 dict 반환:
>
> ```python
> {'cycles': int, 'dram_reads': int, 'dram_writes': int, 'pe_moves': int, 'C': np.ndarray}
> ```
>
> 4×4 PE 에서 N=8 MatMul (8×8 을 4×4 PE 로 tiling) 3 dataflow 실행.

### 단순화 가정

- 모두 float32 (4 byte/elem)
- DRAM read 마다 카운트 (캐싱 무시)
- PE 간 한 hop 이동도 카운트

### 검증 기준

- 3 dataflow 모두 결과 C 가 `np.allclose(A @ B)`
- DRAM access 가 3 dataflow 에서 *서로 다름*
- WS: weight DRAM read 가 가장 적음
- OS: output DRAM write 가 가장 적음
- IS: input DRAM read 가 가장 적음

### 워크로드 변형 (시간 남으면)

| 워크로드 | (M, K, N) | 예상 winner |
| --- | --- | --- |
| Batched FC | (1024, 512, 64) | WS |
| 단발 GEMV | (1, 512, 1024) | IS |
| 큰 균형 | (2048, 2048, 2048) | (균형) |

## Block 3 — 시각화: 3 dataflow 한 그림에 (40분)

## 흔한 막힘 포인트

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| 3 dataflow DRAM 카운트 비슷 | tiling 작아서 reuse 효과 안 보임 | N=8 또는 16, PE 4×4 유지 |
| OS 의 output DRAM write 가 0 | flush 단계 누락 | 시뮬 종료 시 PE partial sum 을 DRAM 쓰기 |
| IS 가 OS 와 거의 같음 | input/output stationary 코드 구분 안 됨 | *PE register 에 무엇이 머무는가* 명시 |
| 시각화 3개 비슷해 보임 | stationary 시각적 강조 부족 | 색깔/박스 두께 차별화 |
